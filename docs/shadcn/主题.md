# shadcn 主题

## Theming

### 用户问题

`https://ui.shadcn.com/docs/theming` 这个是我用的 css 库,帮我解读一下这个

### 问答整理

这份文档的核心只有一句话：`shadcn/ui` 推荐你不要在组件里直接写具体颜色，而是统一写“语义化 token”，然后在全局 CSS 里用变量定义主题。

比如你平时不要优先写 `bg-zinc-900 text-white`，而是写：

```tsx
<div className="bg-background text-foreground" />
<button className="bg-primary text-primary-foreground" />
```

这样你以后改主题，只需要改 CSS 变量，不需要重写组件类名。

#### 你需要理解的 4 个重点

1. `shadcn` 的主题核心是 CSS 变量。
前提是 `components.json` 里：

```json
"tailwind": {
  "cssVariables": true
}
```

这会让 Tailwind 把变量映射成工具类，例如：
- `bg-background`
- `text-foreground`
- `border-border`
- `ring-ring`

2. token 是“语义名”，不是“颜色名”。
例如：
- `background / foreground`：页面默认背景和文字
- `card / card-foreground`：卡片背景和卡片内容
- `primary / primary-foreground`：主操作、主强调
- `muted / muted-foreground`：弱化内容、说明文字
- `accent / accent-foreground`：hover、选中、高亮
- `destructive`：危险操作
- `border / input / ring`：边框、输入框、焦点环

重点是：你写组件时表达“用途”，不是表达“颜色值”。

3. 暗黑模式不是重写组件，而是覆盖同一批 token。
在 `:root` 里定义浅色主题，在 `.dark` 里定义深色主题：

```css
:root {
  --background: ...;
  --foreground: ...;
}

.dark {
  --background: ...;
  --foreground: ...;
}
```

组件还是写：

```tsx
<div className="bg-background text-foreground" />
```

切换 `.dark` 后，整站就自动换皮。

4. `@theme inline` 是把 CSS 变量接到 Tailwind 主题系统上。
例如：

```css
@theme inline {
  --color-background: var(--background);
  --color-primary: var(--primary);
}
```

这样 Tailwind 才能识别 `bg-background`、`bg-primary` 这些类。

#### token 命名规则

它是成对出现的：

- `primary` 表示这个表面的背景色
- `primary-foreground` 表示放在这个表面上的文字/图标色

例如：

```tsx
<div className="bg-primary text-primary-foreground" />
```

意思不是“主色文字”，而是“主色背景上的前景色”。

这套规则适用于：
- `background / foreground`
- `card / card-foreground`
- `popover / popover-foreground`
- `secondary / secondary-foreground`
- `accent / accent-foreground`
- `sidebar-*`

#### radius 这一段的意思

`--radius` 是全局圆角基准值，其他圆角从它推导出来：

- `radius-sm`
- `radius-md`
- `radius-lg`
- `radius-xl`
- `radius-2xl`

所以你后面如果想统一“更圆”或“更硬朗”，只改一个 `--radius`，整套组件会一起变，不要到处写死 `rounded-lg` / `rounded-xl` 的视觉标准。

#### 新增自定义 token 的正确方式

如果你要加品牌语义，比如 `warning`，不是直接乱写颜色类，而是：

1. 在 `:root` 和 `.dark` 里都定义变量
2. 在 `@theme inline` 里暴露给 Tailwind
3. 组件里使用语义类名

例如：

```css
:root {
  --warning: ...;
  --warning-foreground: ...;
}

.dark {
  --warning: ...;
  --warning-foreground: ...;
}

@theme inline {
  --color-warning: var(--warning);
  --color-warning-foreground: var(--warning-foreground);
}
```

之后就能用：

```tsx
<div className="bg-warning text-warning-foreground" />
```

这很重要，因为它决定了你后面扩展设计系统时是否还能保持一致性。

#### 对你后面写代码的直接约束

结合你说的“严格 FSD 学习”，我后面会这样落地：

1. 主题变量定义放全局，不放业务 slice。
也就是放在 `app` 或 `shared` 的全局样式入口里，而不是 `features/*`、`entities/*` 里各自定义颜色体系。

2. 组件里优先使用语义 token，不直接写具体色值。
优先：
- `bg-background`
- `text-foreground`
- `bg-primary`
- `text-muted-foreground`

谨慎使用：
- `bg-red-500`
- `text-zinc-700`

除非是极少量一次性视觉细节。

3. 如果要扩展新颜色语义，先扩 token，再用类名。
不要在业务组件里直接发明一套私有颜色规则。

4. `shared/ui` 里的基础组件应完全依赖 token。
这样它们才能被全局主题驱动，而不是跟某个页面耦合。

#### 你可以把它理解成一句工程规则

`shadcn/ui` 的主题系统本质上是：

- 在全局定义设计 token
- 用 Tailwind 映射成语义类
- 组件只消费语义类
- 明暗主题只改变量，不改组件

如果你愿意，我下一步可以直接结合你当前项目结构，给你定一版“适合 FSD 的主题文件放置方案”，包括：
- `globals.css` 应该放哪些 token
- `shared/ui` 组件应该怎么写 class
- 哪些颜色能直接写，哪些必须走 token

## Next.js Dark Mode

### 用户问题

帮我详细解读这个文章

### 问答整理

这篇文章讲的是：在 `Next.js + shadcn/ui` 里，如何用 `next-themes` 做深浅色主题切换。它不是在讲“怎么改颜色”，而是在讲“怎么让 `.dark` 这个类稳定地挂到页面根节点上”，从而让你前面那套 CSS token 生效。

#### 先说整体机制

你前面那篇 theming 文档里已经有了这套规则：

```css
:root { ...浅色变量... }
.dark { ...深色变量... }
```

这意味着 dark mode 的本质不是改组件代码，而是：

1. 在 HTML 某个祖先节点上加一个 `.dark`
2. 让 CSS 变量切换到深色版本
3. 所有 `bg-background`、`text-foreground` 之类的语义类自动换肤

`next-themes` 做的就是第 1 步：帮你管理“当前主题是什么”，并把对应 class 挂上去。

#### 第一步：安装 `next-themes`

```bash
npm install next-themes
```

这个库的职责很明确：

- 读取和保存用户主题偏好
- 支持 `light / dark / system`
- 在客户端把主题同步到 DOM
- 避免你自己写一堆 `localStorage + matchMedia + classList` 逻辑

也就是说，它不是一个 UI 库，而是一个“主题状态和 DOM 同步器”。

#### 第二步：创建 `ThemeProvider`

文档给的代码是：

```tsx
"use client"

import * as React from "react"
import { ThemeProvider as NextThemesProvider } from "next-themes"

export function ThemeProvider({
  children,
  ...props
}: React.ComponentProps<typeof NextThemesProvider>) {
  return <NextThemesProvider {...props}>{children}</NextThemesProvider>
}
```

这段代码的意义有 3 层。

1. `"use client"` 是必须的。
`next-themes` 依赖浏览器环境，比如：
- `localStorage`
- `window.matchMedia`
- DOM class 切换

这些都只能在 Client Component 里运行，所以这个 provider 必须是客户端组件。

2. 这是一个“项目自己的包装层”。
它没有直接到处使用 `next-themes` 的原始 provider，而是包了一层自己的 `ThemeProvider`。这样做的好处是：
- 以后你改默认配置，只改一个地方
- 以后你想加项目自己的主题逻辑，也有统一入口
- 业务代码只依赖你自己的 provider，不直接耦合第三方库

3. `React.ComponentProps<typeof NextThemesProvider>` 的意思是“继承原组件的全部 props 类型”。
所以你这个包装组件依然能传：
- `attribute`
- `defaultTheme`
- `enableSystem`
- `disableTransitionOnChange`
等所有原始配置，而且类型自动跟着第三方库走。

这是一个很标准的 React 包装写法。

#### 第三步：把 `ThemeProvider` 包到根布局

文档示例：

```tsx
import { ThemeProvider } from "@/components/theme-provider"

export default function RootLayout({ children }: RootLayoutProps) {
  return (
    <>
      <html lang="en" suppressHydrationWarning>
        <head />
        <body>
          <ThemeProvider
            attribute="class"
            defaultTheme="system"
            enableSystem
            disableTransitionOnChange
          >
            {children}
          </ThemeProvider>
        </body>
      </html>
    </>
  )
}
```

这一步是整篇文章最重要的部分。

##### 1. 为什么包在根布局

因为主题是全站级能力，不是某个页面局部能力。

如果你只在某个页面包 provider：
- 其他页面拿不到主题上下文
- 根节点 class 可能不同步
- 切页时主题行为会不一致

所以它必须放在 app 根布局，也就是整个应用最外层的共享壳子里。

按照你前面的 FSD 约束，这类 provider 也确实应该属于 `app` 层，而不是 `features` 或 `pages`。

##### 2. 为什么要 `suppressHydrationWarning`

这是 Next.js/React SSR 场景下的关键细节。

问题背景是这样的：

- 服务端渲染时，服务器并不知道用户浏览器里保存的是 `light` 还是 `dark`
- 客户端挂载后，`next-themes` 才会读取本地设置或系统主题
- 所以服务端输出的 HTML 和客户端第一次计算出的 HTML 可能不完全一致

最常见的不一致点就是 `<html>` 节点上的 class，比如：
- SSR 时没有 `.dark`
- 客户端 hydration 后加上 `.dark`

React 会把这种情况当成 hydration mismatch，并发出警告。

`suppressHydrationWarning` 的意思就是：
“我知道这里服务端和客户端可能不一样，这个差异是有意的，不要因为这个报 hydration 警告。”

注意，这不是“修复不一致”，而是“告诉 React 这是预期行为”。
真正造成不一致的根因，是主题要到客户端才能确定。

##### 3. `attribute="class"` 是什么意思

这个配置决定 `next-themes` 用什么方式把主题写进 DOM。

`attribute="class"` 表示它会通过 class 控制主题，比如：

```html
<html class="dark">
```

这正好匹配你前面 theming 文档里的写法：

```css
.dark {
  --background: ...
  --foreground: ...
}
```

也就是说，这里选择 `class` 是因为你的 CSS 主题就是靠 `.dark` 选择器触发的。

如果不是 `attribute="class"`，而是别的属性方式，比如 `data-theme`，那你的 CSS 就得改写，不再是 `.dark { ... }` 这种形式。

所以这里不是随便配的，而是和整套主题 CSS 强绑定。

##### 4. `defaultTheme="system"` 是什么意思

这个配置表示：
如果用户没有手动选过主题，默认跟随系统设置。

系统设置通常来自浏览器的 `prefers-color-scheme`，例如：
- 系统是深色，就默认 dark
- 系统是浅色，就默认 light

这比强行默认 `light` 或 `dark` 更自然，因为它尊重用户设备偏好。

需要注意的是：
- `defaultTheme="system"` 只决定“初始默认”
- 一旦用户手动切换过，通常会存入本地，下次优先用用户选择

##### 5. `enableSystem` 是什么意思

它表示开启“跟随系统主题”的能力。

和 `defaultTheme="system"` 关系很近，但不是完全一回事：

- `defaultTheme="system"`：初始默认值设为 system
- `enableSystem`：允许 `system` 成为一个合法主题选项，并监听系统变化

如果不开 `enableSystem`，你就不能真正把“系统主题”作为一个动态模式使用。

一般搭配关系就是文档这样写：
- `defaultTheme="system"`
- `enableSystem`

这样主题模式就通常有三种：
- `light`
- `dark`
- `system`

##### 6. `disableTransitionOnChange` 是什么意思

这是个体验优化项。

很多组件会写 CSS transition，比如：
- 背景色渐变
- 边框过渡
- 文字颜色过渡

如果切换主题时不处理，页面所有元素可能同时进行颜色动画，看起来会闪、脏、拖沓。

`disableTransitionOnChange` 的目的就是：
在主题切换瞬间，临时禁用过渡动画，切完再恢复。

这样用户会看到一个干净的“整体换肤”，而不是满屏元素慢慢变色。

这在 shadcn 这类基于 token 的全局主题系统里很实用，因为 token 一变，几乎整页都会跟着变。

#### 第四步：添加 mode toggle

文档最后说要放一个切换按钮。

这一步本身不复杂，本质就是在客户端调用 `next-themes` 提供的状态和设置函数，例如切到：
- `light`
- `dark`
- `system`

但这里有一个你必须理解的点：

**Provider 负责“全局主题能力”，Toggle 负责“用户交互入口”。**

两者分工不同：

- 没有 Toggle，主题系统仍然能工作，只是用户不能手动切换
- 没有 Provider，Toggle 根本无效，因为没有主题上下文可改

所以完整链路是：

1. `ThemeProvider` 建立上下文
2. `next-themes` 读本地/系统偏好
3. 在 `<html>` 上挂 `dark` 或不挂
4. 全局 CSS 变量切换
5. 组件通过语义 token 自动变色
6. Toggle 只是修改当前主题状态

#### 和你当前 shadcn 主题文档的关系

这两篇文档是上下游关系：

前一篇 `Theming` 解决的是：
- 主题 token 怎么定义
- 组件怎么消费 token
- `.dark` 里怎么覆盖变量

这一篇 `Next.js dark mode` 解决的是：
- 谁来控制 `.dark`
- 在 Next.js SSR 下怎么安全挂载
- 用户怎么切换 light/dark/system

你可以把它们串成一句完整的话：

- `globals.css` 定义浅色和深色 token
- `next-themes` 负责给 `<html>` 加上 `.dark`
- 组件只写 `bg-background`、`text-foreground` 这类语义类
- 切主题时，全站自动换肤

#### 放到你的 FSD 里该怎么理解

按严格 FSD，我后面会这样看待这几个东西：

1. `ThemeProvider` 属于 `app` 层。
因为它是应用级 provider，不是业务功能。

2. 全局主题 CSS 也属于 `app` 或 `shared` 的全局基础设施。
不是某个 feature 私有资源。

3. “切换主题”这个用户动作，如果做成按钮组件，通常更接近一个 `feature`。
因为它是一个明确的用户能力，而不是纯基础设施。

也就是说你后面大概率会是这种职责分离：
- `app/providers/theme`：接入 `next-themes`
- `shared/config/theme` 或全局样式：定义 token
- `features/theme-switcher`：主题切换交互
- `shared/ui`：不关心主题逻辑，只消费 token

这套分法和 FSD 是一致的。

#### 这篇文章里最容易忽视的坑

1. `ThemeProvider` 必须是 client component。
漏掉 `"use client"` 会直接出问题。

2. 根节点要加 `suppressHydrationWarning`。
否则很容易出现 hydration warning。

3. 你必须让 CSS 选择器和 provider 配置一致。
既然你用的是 `.dark { ... }`，那就要 `attribute="class"`。

4. 主题切换组件本身通常也需要是 client component。
因为它要读取和修改当前主题状态。

5. 不要在业务组件里写死 `dark:bg-*` 作为主方案。
在 shadcn 体系里，优先应该是 token 驱动。
`dark:*` 可以作为局部补充，但不该取代语义 token 系统。

#### 一句话总结

这篇文章本质上是在告诉你：

在 Next.js 里，`next-themes` 负责把主题状态安全地同步到 `<html class="dark">`，而你前面定义在 `:root` 和 `.dark` 里的 shadcn token 则负责真正完成视觉换肤。

- `ThemeProvider` 放哪
- `globals.css` 怎么组织
- `theme switcher` 放哪一层
一次性定下来。
