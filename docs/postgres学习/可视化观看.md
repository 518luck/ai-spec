# PostgreSQL 可视化层级结构详解

PostgreSQL 采用的是“一个实例（Instance）下包含多个数据库（Database）”的模型，且将“业务数据对象”与“管理对象”进行了明确的分层。

---

## 1. 实例层级（Top Level）

在可视化工具（如 pgAdmin, DBeaver）中，连接到一个 PostgreSQL 实例后，通常会看到以下三大类：

### 📁 数据库 (Databases)
指该实例中承载业务数据的逻辑库列表。
- **postgres**: 默认自带的管理连接库，通常是首次连接时的默认库。
- **业务库**: 用户自行创建的库（如 `test`, `shop`, `app_db`）。

### 👤 管理员 (Administrator / Roles)
存放实例级的全局管理对象，而非具体的业务表。
- **用户 / 角色 (Roles)**: PostgreSQL 将用户统一视为 Role，负责登录与权限控制。
- **表空间 (Tablespaces)**: 定义数据在磁盘上的物理存储位置。
- **权限相关**: 全局权限配置。

### 📊 系统信息 (System Information)
监控数据库服务器本身的运行状态。
- **版本与配置**: 服务器版本号、内存分配等参数。
- **会话/连接**: 当前活跃的连接、后台进程、锁（Locks）信息。
- **统计信息**: 查询性能、存储占用等统计数据。

---

## 2. 数据库层级（Database Level）

当你展开某一个具体的数据库（如 `postgres`）时，其内部结构通常如下：

### 🧱 模式 (Schemas) —— **最核心部分**
模式是数据库内部的一层命名空间或“分类文件夹”。
- **层级关系对比**:
  - **MySQL**: 一个数据库 -> 一堆表
  - **PostgreSQL**: 一个数据库 -> **多个模式 (Schema)** -> 每个模式下的表、视图、函数等。
- **public**: 默认模式。如果你不指定模式，所有的表都会默认放在这里。

### 🧩 扩展 (Extensions)
PostgreSQL 的插件功能。
- 例如：`uuid-ossp`（生成 UUID）、`pgcrypto`（加密函数）、`postgis`（地理信息处理）。

### ⚡ 事件触发器 (Event Triggers)
针对数据库级事件（如 `CREATE TABLE` 等 DDL 操作）的触发器，不同于普通的行级触发器。

### 📦 存储 (Storage)
偏向底层的存储对象管理。

---

## 总结：层级导图

```text
Instance (服务器实例)
├── Administrator (角色、表空间等全局管理)
├── System Info (性能监控、版本参数)
└── Databases (数据库列表)
    ├── Database A
    │   ├── Schemas (模式: public, user_schema...)
    │   │   ├── Tables (表)
    │   │   ├── Views (视图)
    │   │   └── Functions (函数)
    │   ├── Extensions (插件)
    │   └── Triggers (触发器)
    └── Database B
```
