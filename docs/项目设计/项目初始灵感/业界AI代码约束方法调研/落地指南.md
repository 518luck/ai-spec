# 从"设计模式约束"到"规则文件实践"——落地指南

> 基于前面的讨论和调研，给出一份可操作的落地指南。

---

## 一、核心认知

设计模式 → 架构决策规则 → 规则文件 → AI 产出可维护的代码。

你不需要在规则文件中提及任何设计模式的名字。你只需要把**判断条件**写清楚。

---

## 二、写一份好的 AGENTS.md（或 CLAUDE.md）

### 2.1 模板

```markdown
# 项目名称

## 项目结构
- `app/` — 页面路由（Next.js App Router）
- `ui/` — UI 组件
- `lib/swr/` — 数据请求 hooks
- `lib/actions/` — Server Actions
- `lib/hooks/` — 通用 hooks

## 技术栈
- 框架: Next.js 15 (App Router)
- 状态管理: React Context + useState
- 服务端数据: SWR
- 表单: Server Actions
- 样式: Tailwind CSS

## 开发命令
- 开发: `pnpm dev`
- 构建: `pnpm build`
- Lint: `pnpm lint`
- 类型检查: `pnpm typecheck`
- 测试: `pnpm test`

## 架构决策

### 状态管理
| 场景 | 方案 |
|------|------|
| 组件内部状态 | useState |
| 跨组件共享（单步骤） | context 定义在当前组件文件内 |
| 跨组件共享（多步骤） | 独立 context.tsx 文件 |
| 跨页面共享 | URL 参数 |
| 服务端数据 | SWR |

### 文件组织
- 多步骤流程必须抽独立 context 文件
- UI 组件之间不允许互相 import
- 每个 feature 目录下只能有一个入口组件
- 工具函数放 utils.ts

### 代码风格
- 使用 TypeScript strict 模式
- 使用 ES modules (import/export)，不用 CommonJS
- 优先解构导入: `import { foo } from 'bar'`
- 不添加注释，除非被要求

## 常见坑
- 本地开发需要配置 .env.local（参考 .env.example）
- 某些 API 路由需要认证才能访问
```

### 2.2 原则

1. **精简**：每一行都问"删掉这条 AI 就会犯错吗？"
2. **具体**：写命令、写路径、写判断条件，不写空泛的原则
3. **结构化**：用表格和列表，方便 AI 快速检索
4. **提交到 Git**：团队共享，持续迭代

---

## 三、配合 Linter 形成双重保障

```
规则文件 → 告诉 AI "应该怎么做"（预防）
Linter   → 检测 AI "是否做错了"（发现）
```

在规则文件中明确写出 lint 命令，让 AI 在修改代码后自动执行检查。

---

## 四、用现有代码作为最有效的"规则"

```
"参照 register/context.tsx 的结构，为支付流程写一个类似的状态管理"
```

比写 10 行规则文件更有效。AI 的模式复制能力极强，给它好的参考就能产出风格一致的代码。

---

## 五、持续迭代规则文件

规则文件不是一次写好就结束的。每次发现 AI 犯了不该犯的错误：

1. 在规则文件中补充对应的判断条件
2. 如果规则文件已经很长，考虑是否能用 Linter 规则替代
3. 定期清理过时的规则

---

## 六、与设计模式的关系

```
设计模式（知识储备）
  ↓ 影响
你写规则文件的判断力（知道什么是好的）
  ↓ 转化为
规则文件中的判断条件（具体到场景的 if/else）
  ↓ 约束
AI 的代码产出（可维护的代码）
```

设计模式是你脑子里的东西，规则文件是写给 AI 看的东西。两者不需要一一对应，但前者是后者的知识基础。
