# ai-spec 团队空间与资源表设计 v1.0

> **状态**：草案 v1.0
> **设计取向**：复制分离模型——资源默认归属个人，"发布到团队"= 复制一份独立副本，个人原件与团队副本互不影响
> **技术栈**：Prisma + PostgreSQL（多 schema：`auth` / `token` / `public`）
> **参考**：Dub.co 的 Workspace 隔离模型（但本设计为"个人拥有 + 可选发布"场景做了根本性改造）

### 修订记录

|---|---|
| v1.0（本版） | 初版：定义 Workspace / WorkspaceMember / Resource 三张核心表；确立"复制分离"发布模型；覆盖游客/用户/团队三类可见性；不含角色权限矩阵（依赖 RBAC v3）、不含发布审核流 |

## §1. 三类用户与核心需求

### 1.1 用户分类

| 类型 | 登录状态 | 能看到什么 |
|---|---|---|
| **游客** | 未登录 | `visibility = public` 的资源（只读） |
| **用户** | 已登录 | 自己创建的个人资源 + 所加入团队里的资源 |
| **团队空间成员** | 已登录 | 所在团队空间里的全部资源（由成员资格决定） |

> 游客不建表——就是"未登录"的状态，靠 Resource.visibility 字段区分可见性。

### 1.2 核心需求（来自产品确认）

1. **资源默认归属个人**：用户创建的资源先在自己名下，不属于任何团队。
2. **"发布到团队" = 复制分离**：用户主动把资源发布到团队时，复制一份独立副本；之后个人原件与团队副本各自演化、互不影响（类似 Git fork）。
3. **多团队共享**：同一资源可以发布到多个团队（每次都是独立复制）。
4. **部分资源公开**：用户可把个人资源标记为 `public`，游客可见（只读）。

## §2. 核心设计决策：复制分离模型

### 2.1 为什么不照搬 Dub 的 projectId 必填模式

Dub 的模型是"资源天生属于 workspace"，`Resource.projectId` 必填。但本项目的资源**天生属于个人**，只是可选地发布到团队。两者本质不同：

| | Dub 模型 | 本项目（复制分离） |
|---|---|---|
| 资源的"家" | workspace | **个人** |
| 成员能看到什么 | workspace 里所有资源 | 用户**主动发布**的那部分副本 |
| 核心关系字段 | `Resource.projectId`（必填）| `Resource.ownerId`（必填）+ 可选 `workspaceId` |
| "发布"概念 | 无 | **有**（复制一份，写 workspaceId） |

### 2.2 复制分离语义

```
用户的个人资源（原件）
    │
    │  点击"发布到 A 团队" = 复制一份
    ▼
团队 A 里的资源（独立副本）
    │
    │  团队成员编辑副本 → 不影响原件
    │  用户编辑原件 → 不影响副本
    ▼
两份资源各自独立演化
```

**关键结论**：因为团队里的资源是独立副本（不是引用），所以它确实"属于团队"——此时 `workspaceId` 字段是合理的。Resource 表用 `workspaceId` 是否为 null 区分"个人资源"和"团队资源"，一张表承载两种。

### 2.3 不需要 ResourceShare 关系表

常规的"共享"场景需要一张关系表（resourceId + workspaceId）。但本项目是"复制分离"——发布后就是独立的 Resource 记录，不需要维护共享引用关系。`sourceResourceId` 仅用于**溯源**（"这个团队副本从哪个个人原件来"），不建立任何联动。

## §3. 表设计（Prisma schema）

> 以下 schema 放在新的 `prisma/schema/workspace.prisma` 文件，使用 `@@schema("public")`。现有 User 表在 `auth` schema，跨 schema 关联 PostgreSQL 原生支持。datasource 需补充 `schemas = ["auth", "token", "public"]`。

### 3.1 Workspace —— 团队空间

```prisma
// 团队空间（多人共享资源的隔离边界）
model Workspace {
  id          String   @id @default(cuid())
  name        String
  slug        String   @unique          // URL 标识，全局唯一
  description String?  @db.Text
  ownerId     String                    // 创建者（团队所有者）

  // 关联创建者（跨 schema，指向 auth.User）
  owner       User     @relation("WorkspaceOwner", fields: [ownerId], references: [id], onDelete: Cascade)

  // 成员关系（通过 WorkspaceMember 中间表）
  members     WorkspaceMember[]

  // 团队里的资源副本
  resources   Resource[]

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@index([ownerId])
  @@schema("public")
}
```

### 3.2 WorkspaceMember —— 团队成员中间表

```prisma
// 团队成员关系（用户 ↔ 团队空间 的多对多纽带）
model WorkspaceMember {
  id          String   @id @default(cuid())
  userId      String
  workspaceId String

  // 关联用户（跨 schema，指向 auth.User）
  user        User      @relation("MemberUser", fields: [userId], references: [id], onDelete: Cascade)
  workspace   Workspace @relation(fields: [workspaceId], references: [id], onDelete: Cascade)

  // ⚠️ 角色字段本版占位，具体角色值在 v1.2 对接 RBAC v3 后填充
  // 预留：role WorkspaceRole @default(member)

  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  // ⭐ 防重复加入：同一用户在同一团队只能有一条成员记录
  @@unique([userId, workspaceId])
  // ⭐ 加速"列出某团队的所有成员"
  @@index([workspaceId])
  @@schema("public")
}
```

**设计说明：**
- `@@unique([userId, workspaceId])`：防止应用 bug 导致同一用户重复加入同一团队。
- `@@index([workspaceId])`：高频查询是"列出某团队的所有成员"，所以索引 projectId。反查"某用户加入了哪些团队"由复合唯一约束的前缀覆盖，无需单加索引。
- 两侧 `onDelete: Cascade`：删用户或删团队，连带清掉成员关系，避免孤儿数据。

### 3.3 Resource —— 资源（一张表承载个人 + 团队）⭐ 核心

```prisma
// 资源：一张表同时承载"个人资源"和"团队副本"
model Resource {
  id        String   @id @default(cuid())
  ownerId   String                    // 创建者（始终有值，资源归属个人）
  workspaceId String?                 // ⭐ null=个人资源 / 有值=团队资源副本

  title     String
  content   String  @db.Text
  summary   String?

  // 可见性：private=仅创建者可见 / public=游客可见（只读）
  visibility Visibility @default(private)

  // 溯源：团队副本记录"从哪个个人原件复制来"，纯审计用，不建立联动
  sourceResourceId String?

  // 关联创建者（跨 schema，指向 auth.User）
  owner     User     @relation("ResourceOwner", fields: [ownerId], references: [id], onDelete: Cascade)

  // 关联团队（可选；null 表示个人资源）
  workspace Workspace? @relation(fields: [workspaceId], references: [id], onDelete: Cascade)

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  // ⭐ 加速"查我的个人资源"
  @@index([ownerId])
  // ⭐ 加速"查某团队的资源"
  @@index([workspaceId])
  // ⭐ 加速"游客查公开资源"
  @@index([visibility])
  @@schema("public")
}

// 资源可见性枚举
enum Visibility {
  private   // 仅创建者（个人资源）或团队成员（团队副本）可见
  public    // 游客可见（只读）
}
```

**关键字段说明：**

| 字段 | 类型 | 含义 |
|---|---|---|
| `ownerId` | `String`（必填）| 谁创建的。资源所有权始终属于个人，即使是团队副本也记录"是谁发布来的" |
| `workspaceId` | `String?`（可选）| **区分个人/团队的关键**：null = 个人资源，有值 = 团队副本 |
| `visibility` | `Visibility` | private（默认）/ public（游客可见）|
| `sourceResourceId` | `String?`（可选）| 仅团队副本有值，记录"从哪个个人原件复制来"。纯溯源，删原件后此字段不联动（可悬空）|

**两种资源形态对照：**

```
个人资源（原件）：
  { ownerId: U1, workspaceId: null, title: "...", visibility: private }

团队资源（副本，从上面复制来）：
  { ownerId: U1, workspaceId: W1, title: "...", sourceResourceId: 上面那个 id }
```

## §4. "发布到团队"的数据流

```
前提：用户 U1 有个人资源 R1（workspaceId = null）

动作：U1 点击"发布到 A 团队"

执行：
  1. 读取 R1 的全部字段
  2. INSERT 新行 R2：
       ownerId        = U1（仍是 U1 发布的）
       workspaceId    = A
       title/content  = 复制自 R1
       sourceResourceId = R1.id    ← 记录来源
       visibility     = private    ← 团队副本默认不对游客公开
  3. R1 和 R2 是完全独立的资源

后续：
  - 团队成员编辑 R2 → R2 变化，R1 不受影响 ✅
  - U1 编辑 R1 → R1 变化，R2 不受影响 ✅
  - U1 删除 R1 → R2 继续存在（独立资源），R2.sourceResourceId 变悬空（保留或置 null）
```

**"发布到多个团队"**：重复上述流程，每次创建独立副本（R2 给团队 A、R3 给团队 B），互不影响。

## §5. 查询逻辑（三类用户）

### 5.1 游客（未登录）—— 只看公开资源

```sql
SELECT * FROM "public"."Resource"
WHERE visibility = 'public';
```

### 5.2 用户 —— 看"我的个人资源"

```sql
SELECT * FROM "public"."Resource"
WHERE ownerId = '当前用户'
  AND workspaceId IS NULL;
```

### 5.3 用户 —— 看"某团队空间里的资源"

```sql
-- 先确认当前用户是该团队成员
SELECT r.* FROM "public"."Resource" r
WHERE r.workspaceId = '某团队'
  AND EXISTS (
    SELECT 1 FROM "public"."WorkspaceMember" m
    WHERE m.workspaceId = r.workspaceId
      AND m.userId = '当前用户'
  );
```

### 5.4 用户 —— 看"我能看到的全部资源"（个人 + 所有团队）

```sql
SELECT * FROM "public"."Resource"
WHERE ownerId = '当前用户' AND workspaceId IS NULL   -- 我的个人资源
UNION ALL
SELECT r.* FROM "public"."Resource" r
WHERE r.workspaceId IN (
  SELECT workspaceId FROM "public"."WorkspaceMember"
  WHERE userId = '当前用户'
);
```

## §6. 与 Dub 模型的对比

| 维度 | Dub | 本项目 |
|---|---|---|
| Resource 的 workspaceId | 必填 | **可选**（null=个人，有值=团队）|
| 资源所有权 | 团队 | **个人**（ownerId 始终在）|
| "发布"概念 | 无 | **有**（复制一份，写 workspaceId + sourceResourceId）|
| 共享语义 | 归属 | **复制分离**（fork 模型）|
| 公开/私有 | 无 | **有** visibility 字段（游客可见）|
| 删除 workspace | 级联删所有资源 | 级联删团队副本（个人原件不受影响）|
| 中间表 | ProjectUsers | WorkspaceMember（结构一致）|

## §7. 索引设计说明

| 索引 | 服务场景 |
|---|---|
| `Resource @@index([ownerId])` | 查"我的个人资源"（§5.2） |
| `Resource @@index([workspaceId])` | 查"某团队的资源"（§5.3） |
| `Resource @@index([visibility])` | 游客查公开资源（§5.1） |
| `WorkspaceMember @@unique([userId, workspaceId])` | 防重复加入 + 反查"某用户的团队"（前缀覆盖） |
| `WorkspaceMember @@index([workspaceId])` | 查"某团队的成员列表" |

**未加 `@@index([userId])` 的原因**：复合唯一约束 `@@unique([userId, workspaceId])` 自带索引，且 userId 在最左前缀，已覆盖 `WHERE userId = ?` 查询。

## §8. User 表需要补充的字段

现有 `auth.User` 表（`prisma/schema/schema.prisma:17`）需新增关联，以支持上面的关系：

```prisma
model User {
  // ... 现有字段保持不变

  // 新增：反向关系（不产生物理列，仅用于 Prisma 查询）
  ownedWorkspaces  Workspace[]      @relation("WorkspaceOwner")  // 我创建的团队
  memberships      WorkspaceMember[] @relation("MemberUser")      // 我加入的团队
  ownedResources   Resource[]       @relation("ResourceOwner")   // 我创建的资源
}
```

> 注意：`User` 表在 `auth` schema，业务表在 `public` schema，跨 schema 关联需确保 datasource 的 `schemas` 包含两者。

## §9. 删除规则（默认策略）

| 动作 | 结果 | 理由 |
|---|---|---|
| 删除个人原件 R1 | 团队副本 R2 **不受影响** | R2 是独立资源，不依赖 R1 |
| 删除团队副本 R2 | 个人原件 R1 **不受影响** | 同上 |
| `R2.sourceResourceId` 指向的 R1 被删 | R2 该字段悬空 | 保留溯源记录，不级联（可由应用层置 null）|
| 删除 Workspace | 该团队的所有 Resource 副本被级联删 | `onDelete: Cascade`；个人原件不受影响 |
| 删除 User | 其个人资源 + 创建的团队 + 成员关系全删 | `onDelete: Cascade`（详见 v1.1 落地时确认是否需要软删除）|

**复制分离模型的红利**：删除天然隔离，没有 Dub 那种"删 workspace 级联删全部资源"的连锁风险（因为个人原件和团队副本是独立的）。

## §11. 开放问题（待产品确认）

1. **团队副本能否再被发布到其它团队？** 当前设计允许（它有 ownerId，可再复制），需确认是否符合产品预期。
2. **个人原件删除后，团队副本是否需要通知？** 当前设计无通知机制，v1.2 可加 activity log。
3. **团队副本的 visibility 能否独立设为 public？** 当前设计允许（团队副本也有 visibility 字段），即"发布到团队后再公开给游客"。需确认。
4. **同一资源发布到同一团队是否允许重复？** 当前无约束阻止，可在应用层加"已发布"标记，或加 `@@unique([sourceResourceId, workspaceId])` 约束。
