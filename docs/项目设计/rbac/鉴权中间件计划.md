# 鉴权中间件计划

## 核心原则

中间件按**校验规则**拆分，不按功能拆。校验用到什么，才查什么。

## 三类身份

| 身份     | 接入方式                               | 能做什么                                                       |
| -------- | -------------------------------------- | -------------------------------------------------------------- |
| 游客     | 匿名（无 session 无 key）              | 只读公开资源；个人资源靠 `owner_id` 自然隔离（查不到别人数据） |
| 登录用户 | cookies（浏览器）或 API Key（SDK/MCP） | 完整读写个人资源                                               |
| 团队用户 | cookies（仅浏览器，SDK 不接入团队）    | 个人资源 + 团队协作 + 成员/项目管理                            |

## 接口分层（关键）

```
Server Action（"use server"）          → 网页内操作（改邮箱、管 key、团队管理）
                                          天然仅浏览器可调，第三方调不到
                                          内部 await auth() + 业务校验
                                          不需要中间件

Route Handler（app/api/.../route.ts）   → HTTP 接口
                                          个人资源：withPersonal 中间件
                                          团队资源：withTeam 中间件
```

## 中间件清单（只写这两个）

### withPersonal（个人资源 HTTP 接口）

- 路径：`/api/prompts/[id]` 等
- 接入：cookies + SDK/MCP API Key 都支持（走 `resolveContext`）
- 校验：
  1. 身份解析（登录用户 / 游客）
  2. 读操作：任何人放行（数据靠 `owner_id` 隔离）
  3. 写操作：必须登录；SDK 接入还要校验 scope 含 write
  4. 套餐校验（可选，pro 功能用）

### withTeam（团队资源 HTTP 接口）

- 路径：`/api/teams/[teamId]/prompts/[id]` 等
- 接入：**仅 cookies**，SDK/MCP 不接入
- 校验：
  1. 必须登录
  2. 必须是该团队成员（查 TeamMember）
  3. 角色（owner/admin/member）决定能否做成员/项目管理

## 资源逻辑复用（不是接口复用）

个人和团队资源路径不同 → 不同 route.ts → 各挂各的中间件。
但**增删改查业务代码抽成纯函数**，两个 route 都 import：

```
editPrompt()  ← 共用的业务函数（查库、改数据，不含鉴权）
  ▲                    ▲
  │                    │
/api/prompts/[id]     /api/teams/[teamId]/prompts/[id]
  withPersonal          withTeam
```

## 不需要中间件的场景

| 操作                   | 原因                                       |
| ---------------------- | ------------------------------------------ |
| 改邮箱、改密码         | Server Action，天然仅浏览器                |
| 管 API Key（增删改）   | Server Action                              |
| 团队成员管理、项目设置 | Server Action                              |
| 游客浏览公开资源       | 数据靠 `owner_id` / 公开标识过滤，不需鉴权 |

## 数据隔离（替代部分鉴权）

个人资源天然用 `owner_id = session.user.id` 过滤：

- 游客无 user_id → 查不到任何个人资源
- 其他用户 → where 子句限定，查不到别人的

**权限校验只在"写"和"跨身份读"时才需要**，不是每个接口都要。
