## §1. 核心设计理念

> **找到 → 复制 → 粘贴，三步之内必须完成。**

用户的核心场景是"正在和 AI 对话，想快速调出一个常用提示词"。一切设计都服务这条最短路径。列表页的每个可见单元都必须能**一键复制**，不需要点进详情页。

## §2. 信息架构：草稿 vs 收录

### 2.1 Prompt 生命周期

```
┌─────────────────────────────────────────────────┐
│              用户的 Prompt 生命周期               │
│                                                   │
│   随手记 ──→ 草稿箱 ──→ 转正 ──→ 收录库          │
│  (快速捕获)  (临时暂存)        (整理归档)          │
│                                                   │
│   特点：       特点：            特点：             │
│   无需想标题   可无标签          有标题/描述        │
│   纯文本       随时可删          有标签             │
│   即记即用     无版本            有版本控制         │
│                                  可收藏/可统计     │
└─────────────────────────────────────────────────┘
```

### 2.2 草稿与收录的职责划分

| 维度     | 草稿 (Draft)               | 收录 (Record)                |
| -------- | -------------------------- | ---------------------------- |
| 定位     | 便签纸，临时暂存           | 收纳盒，整理归档             |
| 标题     | 可选（默认取内容前 20 字） | 必填                         |
| 描述     | 无                         | 有                           |
| 标签     | 可选                       | 可选（建议有）               |
| 版本     | 无                         | 有（自动记录每次编辑）       |
| 收藏     | 无                         | 有                           |
| 使用统计 | 无                         | 有（use_count / view_count） |
| 可见性   | 仅自己                     | private / public（广场）     |
| 来源     | 手动新建                   | 草稿转正 / 手动新建 / fork   |

### 2.3 关键动线

草稿可以一键"转正"为收录（带上已有内容，补填标题/描述/标签即可）。

## §3. 页面设计

### 3.1 草稿列表页（`/personal/prompt/drafts`）

采用**卡片瀑布流**，因为草稿内容长度不一，卡片更适合快速扫读+复制。

```
┌──────────────────────────────────────────────────────────┐
│  草稿                                    [+ 新建草稿 ⌘N]  │
│  ┌────────────┐  搜索草稿...           筛选标签 ▾        │
└──┴────────────┴───────────────────────────────────────────┘

┌─────────────────────────┐  ┌─────────────────────────┐
│ 你是一个资深前端工程师…  │  │ 帮我把这段代码重构…     │
│                         │  │                         │
│ #前端  #角色扮演         │  │ #重构                   │
│                         │  │                         │
│ 3 分钟前      [⎘复制]    │  │ 1 小时前     [⎘复制]    │
│         [↗ 转正] [⋯]    │  │         [↗ 转正] [⋯]    │
└─────────────────────────┘  └─────────────────────────┘

┌─────────────────────────┐  ┌─────────────────────────┐
│ 总结以下文章的三个…      │  │ （无标题草稿）           │
│                         │  │ 检查这个 SQL 有没有…     │
│ #总结                   │  │                         │
│ 昨天         [⎘复制]    │  │ 昨天         [⎘复制]    │
└─────────────────────────┘  └─────────────────────────┘
```

**设计要点**：

- 每张卡片**首屏即见复制按钮**，点击直接复制全文到剪贴板，toast 提示"已复制"
- 卡片底部"转正"按钮：弹出对话框补填标题/描述/标签后转为收录，原草稿可删除或保留
- 卡片右上角 `⋯` 菜单：编辑 / 删除 / 转正
- 无标题草稿用内容前 20 字 + "…"作为展示标题
- 标签用 Badge 展示，点击标签即筛选

### 3.2 收录列表页（`/personal/prompt/records`）

采用**左列表 + 右详情**的双栏布局，因为收录有标题/描述/版本/统计，详情信息量大，双栏能保持上下文不丢失。

```
┌─────────────────────────────────────────────────────────────┐
│  收录                              [+ 新建 ⌘N]  [📁 从草稿转正] │
│  ┌──────────────────┐  搜索...     排序: 最近使用 ▾  筛选 ▾  │
└──┴──────────────────┴───────────────────────────────────────┘

┌──────────────────────────┐  ┌────────────────────────────────┐
│ 🔍 搜索...               │  │ 前端代码审查专家                │
│                          │  │                                │
│ ★ 收藏 (3)               │  │ 描述：审查 React/TS 代码质量…   │
│   ▸ 前端代码审查专家      │  │                                │
│   ▸ 周报生成器            │  │ ┌────────────────────────────┐ │
│                          │  │ │ 你是一个资深代码审查专家…   │ │
│ 最近使用                 │  │ │                            │ │
│   ▸ SQL 优化助手          │  │ │ {{language}} 代码如下：     │ │
│   ▸ 会议纪要整理          │  │ │ {{code}}                   │ │
│                          │  │ └────────────────────────────┘ │
│ 全部 (12)                │  │         [⎘ 复制完整内容]        │
│   前端代码审查专家  ★     │  │                                │
│   周报生成器        ★     │  │ #前端  #角色扮演  #审查        │
│   SQL 优化助手            │  │                                │
│   会议纪要整理            │  │ 使用 23 次 · 浏览 45 次        │
│   翻译润色                │  │ 版本 v3 · 2 天前更新           │
│   ...                    │  │                                │
│                          │  │ [✏ 编辑] [🕘 版本] [★ 收藏]    │
│ #标签                    │  │ [🔗 复制链接] [⋯ 更多]         │
│  前端 角色 总结 翻译 ...  │  └────────────────────────────────┘
└──────────────────────────┘
```

**设计要点**：

- 左侧列表分组：**收藏置顶** → 最近使用 → 全部，让最高频的 prompt 永远在最前面
- 排序选项：最近使用 / 使用次数最多 / 最近创建 / 最近更新
- 右侧详情区是核心：内容预览 + **大号复制按钮** + 元信息 + 操作区
- 变量占位符（`{{language}}`、`{{code}}`）高亮显示——复制时可选择"填充后复制"（见 §3.4）

### 3.4 复制时的变量填充（进阶体验）

当 prompt 含 `{{变量}}` 时，点复制弹出轻量浮层：

```
        ┌─────────────────────────────┐
        │  填充变量后复制              │
        │                             │
        │  language                   │
        │  ┌─────────────────────┐    │
        │  │ TypeScript          │    │
        │  └─────────────────────┘    │
        │                             │
        │  code                       │
        │  ┌─────────────────────┐    │
        │  │ 粘贴代码...          │    │
        │  │                     │    │
        │  └─────────────────────┘    │
        │                             │
        │  ☑ 记住填入值  [⎘ 直接复制原文]│
        │            [取消]  [复制]   │
        └─────────────────────────────┘
```

- 没有变量时：点复制按钮直接复制，无弹窗
- 有变量时：弹浮层填充，也可选"直接复制原文"跳过
- "记住填入值"：下次复制预填上次的值（存 localStorage 即可）

## §5. 关键设计决策（容易踩坑的点）

1. **"最常用"和"收藏"要分开**——前者是客观数据（use_count 排序），后者是主观标记。用户可能收藏了一个"重要但低频"的 prompt，不能混在一起。数据模型里也分别有 `use_count` 和 `PromptFavorite`，方向是对的。

2. **复制是第一公民操作**——每个能展示 prompt 的地方，都必须有触手可及的复制按钮。建议复制成功后有 toast + 临时视觉反馈（按钮变"已复制 ✓"2 秒）。

3. **草稿要极低门槛**——新建草稿不该有必填项，甚至可以"从剪贴板一键新建"。草稿的意义就是"先记下来再说"。

4. **版本控制要对用户透明**——用户不需要主动"创建版本"，每次编辑保存自动留档就行。`PromptRecordVersion` 的 `message` 字段可让用户选填"改了什么"，不填就自动用"编辑于 {时间}"。

5. **变量占位符是杀手锏**——大多数 prompt 有可变部分（角色名、代码、主题）。`{{变量}}` 语法 + 复制时填充，能避免用户每次手动替换文本。这个功能不大但体验提升显著，建议至少做到 P1。

6. **快捷键**——⌘N 新建、⌘K 搜索面板、⌘C 在选中项上直接复制。面向"快速使用"的产品必须有键盘流。

### 6.2 数据模型

### 6.3 建议实现顺序

1. 先做后端 Server Actions + zod schema（草稿 CRUD → 收录 CRUD → 收藏 → 版本）
