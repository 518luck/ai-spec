# 收录版本上限与压缩方案

> 状态：**待实现**（设计方案已确定，等业务需要时再落地）
> 创建时间：2026-07-24

## 背景

收录（PromptRecord）每次保存（PATCH 带内容变化）会创建一个版本记录（PromptRecordVersion）。
无上限的版本增长会导致：

1. **存储膨胀**：快照版存完整 content（最大 100KB），增量版存 diff JSON（2-5KB），长期累积不可控
2. **深度分页变慢**：版本列表 `skip + take` 分页，版本数过万时深翻页性能下降
3. **diff 重建链变长**：增量版从最近的快照锚点重建，虽然最多回溯 10 个，但快照本身也要遍历

## 方案概述：软上限 + 异步队列压缩

采用**双阈值 + 延迟批量处理**策略，99% 的保存操作零额外开销。

```
版本数 0 ~ 200（软上限）：正常保存，零额外开销
版本数 200 ~ 290（缓冲区）：继续保存，不删不合并
版本数 = 290（硬触发点）：往队列丢压缩任务，异步删 90 个最旧版本
压缩完成后回到 200，重新获得 90 的缓冲空间
```

### 为什么不用其他方案

| 方案 | 否决原因 |
|------|---------|
| FIFO 每次淘汰（超 200 就删最旧 1 个） | 每次保存都多一次 DELETE，虽然轻但不够优雅 |
| 合并压缩（每次保存合并最旧增量） | 每次保存都做 diff 重建 + 合并，热路径太重 |
| 阻止保存（超上限拒绝 PATCH） | 打断用户编辑流程，体验差 |
| 后台定时 Cron 扫描压缩 | 引入定时任务扫描全表，资源浪费 |

## 关键设计

### 阈值参数

| 参数 | 值 | 说明 |
|------|------|------|
| `SOFT_LIMIT` | 200 | 软上限，设计上的"版本保留数量" |
| `HARD_TRIGGER` | 290 | 硬触发点，达到此值往队列丢压缩任务 |
| `COMPRESS_COUNT` | 90 | 压缩时删除的最旧版本数（= HARD_TRIGGER - SOFT_LIMIT） |

### 触发时机

在现有 PATCH 事务末尾（`app/api/prompt/records/[id]/route.ts`），插入版本记录后检查：

```ts
// 版本插入完成后，如果达到硬触发点，往队列丢压缩任务
if (nextVersionNumber === HARD_TRIGGER) {
  await enqueueVersionCompress({
    recordId: id,
    deleteBefore: nextVersionNumber - COMPRESS_COUNT, // 删除 versionNumber ≤ 200 的版本
  });
}
// 事务立即返回，用户无感
```

注意：压缩任务**不在事务内执行**，只入队。用户请求不等压缩完成。

### 压缩任务（队列消费端，复用现有 worker 进程）

```
输入：recordId、deleteBefore（要删除的 versionNumber 上限）

1. 查询该收录 versionNumber ≤ deleteBefore 的所有版本
2. 确保保留的第一个版本是快照锚点（isSnapshot=true）
   - 如果恰好不是快照，向前找到最近的快照保留
   - 原因：删除后，后续增量版需要快照作为重建基准
3. 批量删除确定要淘汰的版本
4. 幂等：任务可重复执行（重复删除 0 行不影响）
```

### 快照锚点保护（重要）

当前快照策略：v1 强制快照，之后每 10 版（v10、v20、...）存快照。

压缩删除时**必须保证保留区间内的第一个版本是快照**，否则它之后的增量版无法重建。

例如：要删除 versionNumber ≤ 200 的版本
- v200 恰好是快照 → 直接删 v1-v199
- v201 是增量（v210 才是下一个快照）→ 删除时跳过 v210 之前最近的一个快照

### 失败处理

- 队列消费失败：自动重试（队列标准机制）
- 重试仍失败：版本数暂时超过 290，用户仍可继续保存（下次触发时再次入队）
- 最坏情况：版本数涨到 ~290 + 保存频率，不影响功能正确性，只影响存储量
