# 设计模式与 AI 编程约束

## 核心结论

设计模式不是直接约束 AI 的工具，而是**约束写规则的你**的知识储备。

```
设计模式（思想）→ 架构决策规则（判断条件）→ 示例文档（参考）→ AI 产出可维护的代码
```

三者缺一不可：

- **没有设计模式** → 规则写不对，因为不知道什么才是"好"
- **没有规则** → AI 不知道什么时候该做什么选择
- **没有示例** → AI 不知道具体怎么落地

---

## 1. 为什么不能直接用设计模式约束 AI？

设计模式是架构层面的指导，不是代码层面的约束。比如你告诉 AI "用观察者模式"，它可以写 Context，也可以写 EventEmitter，也可以写 RxJS——同一个模式，产出质量天差地别。

## 2. 真正有效的是：架构决策表 + 目录结构 + 少量示例

### 2.1 架构决策规则（10-20 条就够了）

只需要写**判断条件**，不需要提设计模式的名字：

| 场景                            | 方案                         | 示例                 |
| ------------------------------- | ---------------------------- | -------------------- |
| 组件内部状态                    | useState                     | 按钮展开/收起        |
| 跨组件共享状态（单步骤）        | context 定义在当前组件文件内 | login-form.tsx       |
| 跨组件共享状态（多步骤/跨页面） | 独立 context 文件            | register/context.tsx |
| 跨页面共享状态                  | URL 参数或 Zustand           | —                    |
| 服务端数据                      | SWR / React Query            | lib/swr/ 下的 hooks  |
| 表单提交                        | Server Actions               | lib/actions/         |

### 2.2 Context 放置规则（示例）

```markdown
## Context 放置规则

- 如果状态只在当前组件及其直接子组件内使用 → context 定义在当前组件文件内
- 如果状态需要跨多个独立组件/页面使用 → context 抽成独立文件
```

### 2.3 目录结构（一次写好，基本不变）

```
app/          → 页面路由
ui/           → UI 组件
lib/swr/      → 数据请求 hooks
lib/actions/  → Server Actions
lib/hooks/    → 通用 hooks
```

## 3. 为什么不需要为每种业务场景写示例？

80% 的代码遵循 20% 的规则。规则覆盖判断逻辑，少量示例覆盖落地参考。AI 最擅长的是**模式复制**——它看项目现有代码就能推断出该怎么做。

## 4. 实际案例：dub 项目中的 register/context.tsx

### 为什么单独抽文件？

注册流程有多步骤（signup → verify），`email`、`password` 需要在多个独立组件之间共享：

```
register/
  ├── context.tsx           ← 只管状态定义和 Provider
  ├── signup-form.tsx       ← 只管注册表单 UI
  ├── signup-email.tsx      ← 只管邮箱输入 UI
  ├── verify-email-form.tsx ← 只管验证码 UI（通过 useRegisterContext() 获取 email/password）
  └── resend-otp.tsx        ← 只管重发验证码
```

### 对比 login-form.tsx

登录是单步骤流程，所有子组件都在 `LoginForm` 组件内部渲染，Context 的 Provider 直接包在组件里就够了，不需要单独抽文件。

### 背后的设计思想

- **关注点分离** → context 只管状态，UI 组件只管渲染
- **观察者模式** → Context 本身就是观察者模式的实现
- **外观模式** → `useRegisterContext()` 提供统一的简化接口

但作者写的时候并不是"我要用这三个模式"，而是遇到具体问题，自然选择了这种方案。设计模式是**事后总结**，不是**事前指导**。

## 5. 代码审查兜底

无论 AI 多智能，人必须审查。审查时关注三个问题：

1. 状态放在了正确的层级吗？
2. 组件之间的依赖关系清晰吗？
3. 如果需求变了，改动范围小吗？

这三点本质上就是设计模式关心的东西，但不需要用模式的名字去审查。

---

## 总结

> 设计模式 → 影响你的思维 → 你写出规则 → 规则约束 AI → AI 产出可维护的代码

设计模式不直接约束 AI，但它约束了写规则的你。用设计模式作为指导思想，细化为架构决策规则，再配合少量示例文档，就能让 AI 产出可维护的代码。
