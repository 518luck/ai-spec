# 业界约束 AI 生成代码的方法调研

> 基于 Claude Code、Cursor、VS Code Copilot、OpenCode 等主流 AI 编程工具的官方文档和实践整理。

---

## 一、方法总览

业界约束 AI 生成代码的方法可以分为 **5 个层级**，从软约束到硬约束逐步加强：

```
第 1 层：Prompt 规则文件（软约束，AI 自觉遵守）
第 2 层：代码模板与示例（模式引导）
第 3 层：静态分析工具（自动检测违规）
第 4 层：Hooks 与 CI 自动化（强制执行）
第 5 层：人工审查（最终兜底）
```

---

## 二、第 1 层：Prompt 规则文件

这是目前最广泛使用的方法。所有主流 AI 编程工具都支持通过项目内的规则文件来约束 AI 行为。

### 2.1 各工具的规则文件方案

| 工具 | 规则文件 | 说明 |
|------|---------|------|
| **Claude Code** | `CLAUDE.md` | 支持项目级、用户级、子目录级；可引用外部文件；支持 Skills 机制 |
| **Cursor** | `.cursor/rules/*.md` | 支持全局和项目级规则；支持文件类型匹配 |
| **VS Code Copilot** | `.github/copilot-instructions.md` | 支持项目级指令和 `*.instructions.md` 文件类型规则 |
| **OpenCode** | `AGENTS.md` | 兼容 Claude Code 的 `CLAUDE.md`；支持 `opencode.json` 中引用外部文件和远程 URL |
| **Windsurf/Codeium** | `.windsurfrules` | 类似 Cursor 的规则文件 |

### 2.2 规则文件的最佳实践（来自 Claude Code 官方）

Anthropic 官方文档明确建议：

**应该包含的内容：**
- AI 无法猜到的 Bash 命令（构建、测试、lint）
- 与默认不同的代码风格规则
- 测试指令和偏好的测试运行器
- 仓库规范（分支命名、PR 约定）
- 项目特定的架构决策
- 开发环境特殊配置（必要的环境变量）
- 常见的坑和非显而易见的行为

**不应该包含的内容：**
- AI 通过读代码就能推断的东西
- 标准语言约定（AI 已经知道）
- 详细 API 文档（应该链接而不是内联）
- 频繁变化的信息
- 逐文件的代码库描述
- "写干净代码"这类不言自明的实践

**关键原则：**
> "对于每一行规则，问自己：'删掉这条 AI 就会犯错吗？' 如果不会，就删掉它。臃肿的规则文件会导致 AI 忽略你真正的指令！"

### 2.3 规则文件的分层策略

Claude Code 和 OpenCode 都支持多层级规则：

```
~/.claude/CLAUDE.md                    ← 用户全局规则（所有项目生效）
~/config/opencode/AGENTS.md            ← OpenCode 全局规则
项目根目录/CLAUDE.md                    ← 项目级规则（提交到 Git，团队共享）
项目根目录/CLAUDE.local.md             ← 项目个人规则（不提交）
子目录/CLAUDE.md                       ← 子目录级规则（按需加载）
```

### 2.4 外部文件引用

OpenCode 支持在 `opencode.json` 中引用外部规则文件和远程 URL：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "instructions": [
    "CONTRIBUTING.md",
    "docs/guidelines.md",
    ".cursor/rules/*.md",
    "https://raw.githubusercontent.com/my-org/shared-rules/main/style.md"
  ]
}
```

Claude Code 支持 `@path/to/file` 语法在 CLAUDE.md 中引用其他文件。

## 三、第 2 层：代码模板与示例（模式引导）

### 3.1 Skills 机制

Claude Code 和 OpenCode 都支持 Skills（技能）机制——将领域知识封装成可复用的规则包：

```
.claude/skills/
  ├── api-conventions/SKILL.md     ← REST API 设计规范
  ├── security-review/SKILL.md     ← 安全审查流程
  └── fix-issue/SKILL.md           ← 修复 GitHub Issue 的工作流
```

Skills 的核心思想：
- **按需加载**：AI 只在相关任务时读取，不会在每次对话中都占用上下文
- **可复用**：跨项目共享
- **可参数化**：支持 `$ARGUMENTS` 变量

### 3.2 给 AI 提供具体参考

Claude Code 官方最佳实践建议：

> **"Reference existing patterns. Point Claude to patterns in your codebase."**

```
"参照 register/context.tsx 的结构，为支付流程写一个类似的状态管理"
```

这比任何抽象的设计规则都有效——AI 最擅长的是模式复制。

### 3.3 Custom Agents（自定义子代理）

Claude Code 支持定义专门的子代理：

```
.claude/agents/
  ├── security-reviewer.md     ← 安全审查专员
  ├── test-writer.md           ← 测试编写专员
  └── code-reviewer.md         ← 代码审查专员
```

每个代理有自己的工具权限、模型偏好和行为定义。子代理在独立的上下文中运行，不污染主对话。

## 四、第 3 层：静态分析工具（自动检测违规）

### 4.1 Linter 和 Formatter

这是最成熟、最可靠的约束层。常见工具：

| 语言 | Linter | Formatter |
|------|--------|-----------|
| JavaScript/TypeScript | ESLint | Prettier |
| Python | Ruff, Pylint | Black, Ruff |
| Go | go vet, staticcheck | gofmt |
| Java | Checkstyle, SpotBugs | Google Java Format |
| Rust | clippy | rustfmt |

**关键实践：** 在 AGENTS.md / CLAUDE.md 中明确写出 lint 和格式化命令：

```markdown
## 代码检查
- 修改代码后必须运行: `npm run lint`
- 格式化: `npm run format`
- 类型检查: `npm run typecheck`
```

### 4.2 TypeScript 类型系统

TypeScript 本身就是一种约束。强类型可以在编译期捕获大量 AI 生成的错误。

### 4.3 自定义 ESLint 规则

对于架构级别的约束（如"跨组件状态必须抽独立 context"），可以编写自定义 ESLint 规则来强制执行。

## 五、第 4 层：Hooks 与 CI 自动化（强制执行）

### 5.1 Hooks 机制

Claude Code 支持在特定生命周期节点运行脚本：

```json
// .claude/settings.json
{
  "hooks": {
    "postFileEdit": "npx eslint --fix $FILE"
  }
}
```

> "Hooks 确定性地保证动作发生，不像 CLAUDE.md 指令是建议性的。"

OpenCode 也支持类似的 Hooks 和权限系统。

### 5.2 Pre-commit Hooks

使用 Husky + lint-staged 在提交前自动检查：

```json
{
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"]
  }
}
```

### 5.3 CI 流水线中的 AI 代码审查

根据 Semaphore 的实践，可以在 CI 中加入 AI 审查步骤：

1. **文件选择**：用 `git diff` 获取变更文件
2. **AI 审查**：将变更文件发送给 AI API 进行审查
3. **结果解析**：解析 AI 返回的结构化结果，决定是否通过

### 5.4 权限控制

Claude Code 和 OpenCode 都支持精细的权限控制：

- **Auto 模式**：AI 可以自主执行，但有安全分类器阻止危险操作
- **白名单**：允许特定安全命令（如 `npm run lint`、`git commit`）
- **沙箱模式**：限制文件系统和网络访问

## 六、第 5 层：人工审查（最终兜底）

### 6.1 Claude Code 官方的审查建议

> "Always provide verification (tests, scripts, screenshots). If you can't verify it, don't ship it."

核心审查清单：
1. 状态放在了正确的层级吗？
2. 组件之间的依赖关系清晰吗？
3. 如果需求变了，改动范围小吗？

### 6.2 Writer/Reviewer 模式

Claude Code 官方推荐的并行审查模式——用两个独立的 AI 会话：

| 会话 A（编写者） | 会话 B（审查者） |
|-----------------|-----------------|
| 实现功能 | 审查实现的代码，寻找边界情况和竞态条件 |

### 6.3 先探索，再规划，再编码

Claude Code 官方推荐的工作流：

```
Plan 模式（探索） → Plan 模式（规划） → Build 模式（实现） → Build 模式（提交）
```

> "让 Claude 直接跳到编码可能会产出解决错误问题的代码。"

## 七、关键洞察：上下文管理是最重要的约束

所有 AI 编程工具的官方文档都强调一个核心约束：

> **Claude 的上下文窗口会快速填满，性能随填充而下降。上下文窗口是需要管理的最重要资源。**

这意味着：
- `/clear` 在不相关的任务之间重置上下文
- 用子代理做调研，保持主对话干净
- CLAUDE.md 不要写太长（否则 AI 会忽略真正重要的规则）
- 连续两次纠正失败后，清空上下文重新开始

---

## 八、综合建议

基于以上调研，约束 AI 生成代码的最佳实践是：

```
1. 写一份精简的规则文件（CLAUDE.md / AGENTS.md）
   - 只写 AI 猜不到的东西
   - 包含构建、测试、lint 命令
   - 包含架构决策和项目约定

2. 用现有代码作为示例
   - 让 AI 参照项目中已有的模式
   - 比抽象规则更有效

3. 配置 Linter + Formatter
   - 自动化地捕获格式和风格问题
   - 在 AGENTS.md 中写明命令，让 AI 执行

4. 在 CI 中加入自动化检查
   - Pre-commit hooks 确保基本质量
   - 可选：AI 审查作为额外保障

5. 人工审查
   - 关注架构层面的问题
   - 利用 Writer/Reviewer 模式
```
