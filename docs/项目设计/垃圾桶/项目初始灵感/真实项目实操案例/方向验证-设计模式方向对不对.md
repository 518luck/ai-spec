# 从真实案例看：你走的设计模式方向对不对？

> 基于对 10+ 个知名开源项目的 AGENTS.md / CLAUDE.md 的分析，回答这个核心问题。

---

## 直接回答：方向对，但焦点需要调整

你的核心思路是对的——**用结构化的思想来约束 AI 产出可维护的代码**。但从真实案例来看，业界没有人直接用"设计模式"这个词来约束 AI。他们做的是更具体的事情。

---

## 你和他们之间的差距

### 你目前在想的

```
设计模式 → 架构决策规则 → 规则文件 → 约束 AI
```

这个链条是成立的，但过于抽象。

### 他们实际在做的

看完 Next.js、Cal.com、pnpm、n8n、Elasticsearch 这些项目的规则文件后，你会发现他们的共同特点是：

**1. 不写抽象原则，写具体的命令和路径**

你写：

> "跨组件共享状态必须抽独立 context 文件"

他们写：

> "You must update the proto conversion layer in THREE places or the provider will silently reset to Anthropic:
>
> 1. `proto/cline/models.proto` - Add to the `ApiProvider` enum
> 2. `convertApiProviderToProto()` - Add case mapping
> 3. `convertProtoToApiProvider()` - Add case mapping"

区别在于：**他们写的是"改什么文件、执行什么命令、避免什么坑"，而不是"遵循什么模式"。**

**2. 大量记录已知的坑**

> "Do not use `instanceof Error`. Jest runs tests in a VM context where `instanceof` checks can fail across realms." — pnpm

> "When cancelled mid-operation, the status stays 'generating' forever — no one updates it." — Cline

这些不是设计模式，是**实战经验的结晶**。每个项目都有自己的"部落知识"，规则文件最大的价值就是记录这些东西。

**3. 明确的验证步骤**

几乎所有项目都写了具体的验证命令：

```
1. pnpm lint-fix
2. pnpm test run <test_file.ts>
3. pnpm check
4. pnpm build
```

他们不是告诉 AI "写好代码"，而是告诉 AI "改完之后跑哪些命令来验证"。

**4. 分层决策框架（Cal.com 的做法）**

| 类别          | 规则         |
| ------------- | ------------ |
| **Always do** | 必须做的事   |
| **Ask first** | 拿不准就问人 |
| **Never do**  | 绝对禁止     |

这比"遵循 XX 设计模式"具体得多、可执行得多。

---

## 你的方向应该怎么调整

### 不要丢掉设计模式，但要转化它

设计模式的价值不变——它影响你的判断力，帮你识别"好代码"。但在写给 AI 的规则中，需要这样转化：

| 设计模式思想 | 转化为具体规则                                                         |
| ------------ | ---------------------------------------------------------------------- |
| 关注点分离   | "context 只管状态，不含 JSX；UI 组件只管渲染，不直接操作 localStorage" |
| 外观模式     | "新增 API Provider 时必须修改这三个文件（列举路径）"                   |
| 策略模式     | "多步骤流程用 `step` 字段切换，不要用多个独立的状态变量"               |
| 观察者模式   | "子组件通过 `useXxxContext()` 获取状态，不允许 props 超过 2 层传递"    |
| 单一职责     | "每个文件不超过 300 行；超过就拆分"                                    |

**关键转变：从"用什么模式"变成"具体怎么做"。**

### 补充你目前缺少的东西

从真实案例来看，你的规则文件还应该包含：

**1. 验证命令（所有项目都有）**

```markdown
## 开发流程

1. 修改代码后运行 `pnpm lint`
2. 运行 `pnpm typecheck`
3. 运行相关测试：`pnpm test -- <文件名>`
```

**2. 已知的坑（Cline、pnpm 的做法）**

```markdown
## 常见陷阱

- 不要用 `instanceof Error` 做 Jest 测试中的类型检查
- `useLocalStorage` 返回的值每次渲染都是最新的，用 `useRef` 冻结初始值
- Next.js App Router 中 `searchParams` 是 Promise，需要 `await`
```

**3. Do/Don't 清单（Cal.com 的做法）**

```markdown
## Do

- 使用 `import type { X }` 导入类型
- 使用 early return 减少嵌套
- 注释只写 why，不写 what

## Don't

- 禁止使用 `as any`
- 禁止组件之间互相 import
- 禁止修改自动生成的文件
```

**4. 拿不准就问人（Cal.com 的做法）**

```markdown
## Ask First

- 新增依赖
- 修改数据库 schema
- 删除文件
- 修改公共 API
```

---

## 最终结论

你的方向是对的——**用结构化的思想约束 AI**。但设计模式是底层的知识储备，不是直接写给 AI 的规则。

从业界的实操来看，一份好的规则文件长这样：

```
项目结构（目录说明）
+ 具体命令（构建、测试、lint）
+ 架构决策（状态管理、文件组织的判断条件）
+ Do/Don't 清单
+ 已知的坑
+ 拿不准就问人的分类
```

这些内容背后都有设计模式的思想在支撑，但呈现给 AI 的是**具体的判断条件和操作步骤**，而不是模式的名字。

**设计模式是你的内功，规则文件是外显的招式。内功深厚，招式才能精准。**
