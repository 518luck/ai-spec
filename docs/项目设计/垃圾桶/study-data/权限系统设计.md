# 权限系统设计

## 文档目标

这份文档不是在追求“最省事上线”，而是在追求一套适合学习、并且后续可扩展到真实项目的权限设计。

目标：

- 先把 RBAC 和 ABAC 的边界讲清楚
- 给出适合这个 Prompt/Agent 网站的第一版权限模型
- 给出 Prisma 层的表结构方向
- 给出代码层的统一权限判断方式
- 保证后续新增 Agent、Workspace、审核流时不需要推翻大部分实现

---

## 一、先统一概念

### 1. RBAC 是什么

RBAC 是 `Role-Based Access Control`，中文是基于角色的访问控制。

它的核心是：

- 用户被分配到一个或多个角色
- 权限被分配给角色
- 用户通过角色继承权限

对这个项目来说，RBAC 负责回答：

- 这个用户从“大方向上”有没有资格做某类操作

例如：

- `guest` 可以读公开内容
- `creator` 可以创建 Prompt
- `admin` 可以发布和审核内容

### 2. ABAC 是什么

ABAC 是 `Attribute-Based Access Control`，中文是基于属性的访问控制。

它的核心是：

- 不只看角色
- 还要看用户属性、资源属性、动作属性、环境属性

对这个项目来说，ABAC 负责回答：

- 这个用户在“当前这条具体数据”和“当前这个状态”下，到底能不能操作

例如：

- `creator` 不等于能编辑所有 Prompt
- 只能编辑 `authorId === user.id` 的 Prompt
- `published` 状态的 Prompt 可能不能被作者直接删除
- `private` 内容默认只有 owner 能看

### 3. 这个项目适合什么

这个项目最适合：

- 用 RBAC 做基础授权
- 用 ABAC 做细粒度约束

换句话说：

- RBAC 管“你大体能做什么”
- ABAC 管“你在这条数据上此刻能不能做”

这也是更贴近真实项目的设计。

### 4. ownership 算什么

像“只能编辑自己的内容”这种规则，本质上已经不只是纯 RBAC。

它更接近：

- ABAC 中的资源属性判断
- 或者关系判断

在这个项目里，可以先把它纳入 ABAC 规则统一处理，不必单独再引入一整套 ReBAC 模型。

## 二、这个项目的权限设计原则

这部分应该当成实现时必须遵守的原则。

### 1. 最小权限

默认只给完成任务所需的最小权限，不因为“以后可能要用”就提前放开。

### 2. 默认拒绝

如果一条请求没有明确命中允许规则，就拒绝。

不要做“没配就是允许”。

### 3. 每次请求都校验

权限校验不能只做前端按钮隐藏。

前端只能改善体验，真正的安全校验必须发生在服务端接口层。

### 4. 统一权限入口

不要把权限判断散落在页面、接口和 service 里。

必须统一收口到：

- `hasPermission(user, permission)`
- `canReadPrompt(user, prompt)`
- `canEditPrompt(user, prompt)`

这种函数入口。

### 5. 角色和资源规则分层

不要把这两类规则混在一起：

- 角色拥有什么基础权限
- 资源在什么状态下允许什么动作

前者属于 RBAC，后者属于 ABAC。

## 三、第一版资源和动作

### 1. 核心资源

第一版建议先聚焦这些资源：

- `Prompt`
- `Agent`
- `Tutorial`
- `Folder`
- `TestSession`
- `User`

这几个最适合先建成真正的资源模型。

### 2. 扩展资源

这两个建议先作为第二阶段：

- `Favorite`
- `Workspace`

原因：

- `Favorite` 更像关联关系，而不是复杂业务主体
- `Workspace` 容易引入团队成员、空间角色、共享边界，复杂度高很多

### 3. 第一版动作

第一版动作先定义为：

- `read`
- `create`
- `update`
- `delete`
- `publish`
- `favorite`
- `share`
- `manage`

说明：

- `manage` 现在是占位动作
- 后面项目变复杂后，`manage` 应拆成更明确的动作，例如 `review`、`recommend`、`archive`、`assignTag`

---

## 四、角色设计

第一版角色建议只保留四个：

- `guest`：未登录用户
- `user`：普通登录用户
- `creator`：内容创作者
- `admin`：管理员

说明：

- `guest` 可以不存进数据库，通常是运行时身份
- `user`、`creator`、`admin` 是正式角色
- 后续如果要做运营、审核员、超级管理员，再扩角色即可

---

## 五、权限命名建议

权限不要命名成模糊的大词，建议统一使用：

`资源.动作`

例如：

- `prompt.read`
- `prompt.create`
- `prompt.update`
- `prompt.delete`
- `prompt.publish`
- `agent.read`
- `agent.create`
- `tutorial.read`
- `folder.update`
- `testSession.read`

这样做的好处：

- 一眼能看懂
- 好查表
- 好写代码
- 后面新增资源时命名规则不变

## 七、ABAC：资源 -> 状态 -> 可执行动作

这一层回答：

- 在一条具体资源上，当前状态和属性是否允许当前动作

第一版先以 `Prompt` 作为样板。

### 1. Prompt 建议字段

- `authorId`
- `visibility`
- `status`

建议的枚举：

- `visibility`: `public | private | unlisted`
- `status`: `draft | pending_review | published | archived`

### 2. Prompt 状态规则表

| 状态                 | 作者可读 | 作者可改 | 作者可删 | 其他登录用户可读 | guest 可读 | admin 可管 |
| -------------------- | -------- | -------- | -------- | ---------------- | ---------- | ---------- |
| draft                | 是       | 是       | 是       | 否               | 否         | 是         |
| pending_review       | 是       | 否       | 否       | 否               | 否         | 是         |
| published + public   | 是       | 否       | 否       | 是               | 是         | 是         |
| published + unlisted | 是       | 否       | 否       | 知道链接时可读   | 否         | 是         |
| published + private  | 是       | 否       | 否       | 否               | 否         | 是         |
| archived             | 是       | 否       | 否       | 否               | 否         | 是         |

说明：

- `creator` 的“可更新”并不代表能更新任意 Prompt，只代表具备基础资格
- 最终允许更新需要同时满足：
  - 角色有 `prompt.update`
  - 当前用户是作者，或是管理员
  - 当前状态允许更新

### 3. 第一版 ABAC 判断公式

以 `canEditPrompt(user, prompt)` 为例：

返回 `true` 的建议条件：

- 用户已登录
- 用户有 `prompt.update`
- `user.id === prompt.authorId` 或 `user` 拥有管理员权限
- `prompt.status === draft`

以 `canReadPrompt(user, prompt)` 为例：

可以按下面思路判断：

- 如果 `prompt.status !== published`
  - 只有作者和管理员可读
- 如果 `prompt.status === published`
  - `visibility === public`：所有人可读
  - `visibility === unlisted`：登录态策略可后续再细化
  - `visibility === private`：只有作者和管理员可读

## 八、数据库设计建议

如果你是为了学习完整实现，第一版建议真的把 RBAC 基础设施建出来。

### 1. RBAC 基础表

建议包含：

- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`

说明：

- `users` 和 `roles` 是多对多
- `roles` 和 `permissions` 也是多对多
- 不建议第一版只在 `users` 表里放一个 `role` 字段就结束
- 如果只是为了最快上线，`users.role` 能跑，但不适合你现在这个学习目标

### 2. 业务资源表必须预留的字段

像 `Prompt`、`Agent`、`Tutorial` 这类内容型资源，建议统一预留：

- `authorId`
- `visibility`
- `status`
- `createdAt`
- `updatedAt`

这样做的作用：

- `authorId` 支撑 owner 判断
- `visibility` 支撑公开/私有访问规则
- `status` 支撑草稿、审核、发布、归档流转

### 3. Favorite 不建议做成复杂权限主体

`Favorite` 第一版更适合做成关系表。

例如：

- `userId`
- `targetType`
- `targetId`

如果后面需要收藏夹、排序、标签、备注，再升级成更完整模型。

### 4. Workspace 建议第二阶段再做

`Workspace` 一旦引入，通常会扩展出：

- `workspace_members`
- `workspace_roles`
- 资源归属到 `workspaceId`

这会明显提高权限复杂度。

因此第一版不建议把团队空间和全站内容权限一起做。

## 九、Prisma 落位建议

### 1. schema 放哪里

Prisma 官方当前文档说明：

- 默认 schema 文件是 `prisma/schema.prisma`
- 也支持多文件 schema

对这个项目，推荐：

```text
prisma/
  schema.prisma
  migrations/
  models/
    auth.prisma
    prompt.prisma
    agent.prisma
    tutorial.prisma
```

### 2. 为什么不放进 `src/entities`

原因是：

- Prisma schema 属于基础设施和数据库建模
- FSD 的 `entities` 是前端业务实体切片
- 这两者有关联，但不是同一层

所以：

- 数据库 schema 放 `prisma/`
- 业务实体代码放 `src/entities/`

### 3. Prisma 文件组织建议

如果后面模型越来越多，按域拆分 `.prisma` 文件是合理的。

建议：

- `auth.prisma` 放 `User / Role / Permission / UserRole / RolePermission`
- `prompt.prisma` 放 `Prompt`
- `agent.prisma` 放 `Agent`
- `tutorial.prisma` 放 `Tutorial`

不要按页面拆 Prisma 文件。

## 十、代码层怎么组织

按 FSD 思路，权限逻辑不要散落在页面组件里。

建议目录：

```text
src/
  entities/
    prompt/
      model/
      server/
        repository.ts
    agent/
      model/
      server/
        repository.ts

  shared/
    lib/
      auth/
      permission/
        has-permission.ts
        prompt-access.ts
        agent-access.ts
```

### 1. `hasPermission`

负责 RBAC 基础判断。

例如：

- 用户是否拥有 `prompt.update`

### 2. `canXxx`

负责 RBAC + ABAC 联合判断。

例如：

- `canReadPrompt(user, prompt)`
- `canEditPrompt(user, prompt)`
- `canDeletePrompt(user, prompt)`

### 3. 页面和接口怎么用

前端页面：

- 用 `canXxx` 控制按钮是否显示

服务端接口：

- 用 `canXxx` 做真正拦截

不要只在前端判断。

## 十一、推荐的实现顺序

你现在最好的下手方式不是先写很多接口，而是按顺序做：

### 第一阶段：把规则写清楚

先完成：

- 资源清单
- 动作清单
- 角色表
- Prompt 的状态规则表

### 第二阶段：先建 RBAC 基础设施

先建：

- `roles`
- `permissions`
- `user_roles`
- `role_permissions`

### 第三阶段：先只做 Prompt 这一条完整链路

先实现：

- 创建 Prompt
- 编辑 Prompt
- 查看 Prompt
- 发布 Prompt

### 第四阶段：把权限收口成统一函数

至少先有：

- `hasPermission`
- `canReadPrompt`
- `canEditPrompt`
- `canPublishPrompt`

### 第五阶段：再复制模式到 Agent 和 Tutorial

不要一开始同时做所有资源。

先把一条链路做通，你会学得更快。

---

## 十二、第一版是否会影响后续扩展

不会，只要满足这几个前提：

- 不把权限写死在页面和接口里
- 资源表一开始就有 `authorId / visibility / status`
- 角色权限和资源状态规则分开
- 始终通过统一权限函数判断

这样后面新增：

- `Agent`
- `Workspace`
- 审核流
- 更多角色

通常都是扩展，不是推翻重写。

---

## 十三、当前项目的结论

对于这个 Prompt/Agent 网站，最合适的第一版是：

- 数据库层做完整 RBAC 基础设施
- 业务资源层预留 ABAC 所需字段
- 代码层统一用 `hasPermission + canXxx`
- 先只拿 `Prompt` 跑通完整权限链路

这套做法既适合学习，也适合以后继续扩项目。
