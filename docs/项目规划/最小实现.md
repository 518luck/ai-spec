Prompt/Agent 网站项目架构建议
总体思路
如果一级导航是：
● Prompt
● Agent
● 测试
● 教程
● Workspace
那项目架构最好不要按“页面随便堆”，而要按业务域拆。
否则后面很容易出现这些问题：
● 同一个 PromptCard 到处复用但难以维护
● Tag、Favorite、Ranking 分散在不同页面里
● 组件边界越来越乱
● 后续扩展排行榜、搜索、文件夹时改动成本很高
更稳的做法是按产品能力和业务模块组织代码。
一、按业务模块分层
顶层按“产品能力”拆，不按零碎组件拆。
建议包括：
● prompt
● agent
● playground 或 test
● learn
● workspace
● shared
● entities
各自职责可以这样理解：
● prompt：Prompt 列表、详情、排行、筛选
● agent：Agent 列表、详情、分类、排行
● playground/test：测试台、运行区、变量填写、结果面板
● learn：教程列表、教程详情、专题页
● workspace：收藏、文件夹、草稿、历史、私有内容
● shared：通用 UI、layout、hooks、utils
● entities：跨模块复用的数据对象，例如 prompt、agent、tag、user、folder

如果是 React / Next.js，建议采用这种结构：

这个结构的逻辑是：
● app/：路由层
● features/：页面中的业务功能块
● entities/：核心业务对象
● shared/：真正通用的能力
这是一种比较稳的做法，适合中长期扩展。

四、组件组织方式
不要一开始就建一个很大的 components/ 目录，否则后面很容易失控。
更好的规则是：
● 通用组件放 shared/ui
● 业务组件放各自模块里
例如：
features/prompt/components/prompt-card.tsx
features/prompt/components/prompt-filter.tsx
features/agent/components/agent-card.tsx
features/test/components/test-runner.tsx
shared/ui/button.tsx
shared/ui/dialog.tsx
shared/ui/input.tsx
判断规则：
● 只服务某个业务模块的，放模块内
● 多个模块都复用的，放 shared/ui
五、数据模型要先独立
这个项目很依赖内容模型，所以最好尽早把实体定清楚。
至少应有这些对象：
● Prompt
● Agent
● Tutorial
● Tag
● Folder
● User
● Favorite
● RunRecord 或 TestSession
建议把每个实体的类型、字段、转换逻辑收在一起，例如：
entities/prompt/model/types.ts
entities/prompt/model/schema.ts
entities/prompt/api/get-prompts.ts
entities/prompt/ui/prompt-badge.tsx
这样后面加后端、缓存、排序、搜索时会更顺。
六、测试页要单独当一个系统
测试 这个导航不要只当普通页面，它本质上是一个小工具系统。
建议内部再拆成这些部分：
● 输入面板
● 变量面板
● Prompt/Agent 选择器
● 结果面板
● 历史记录
● 版本切换
也就是说，在架构上它最好是一个完整模块，不要塞进 prompt 或 agent 下面。
七、搜索、标签、排行榜适合作为横切能力
这些功能不是某一个页面独有的，最好单独做 feature：
● search
● ranking
● tag
● folder
● favorite
原因是：
● Prompt 要搜索
● Agent 要搜索
● Workspace 也可能要搜索
如果把这些能力硬写进某个页面，后面复用会很麻烦。
八、权限和状态管理要早点定
至少先明确这些状态应该放在哪里：
● 当前登录用户
● 当前筛选条件
● 当前测试会话
● 收藏状态
● 文件夹状态
通常建议：
● 页面局部状态：组件内
● 跨页面筛选：URL query
● 用户状态：全局 store 或 session
● 测试记录：服务端 + 本地缓存
特别是筛选条件，尽量放在 URL 里，不然分享和回退体验会很差。
九、能落地的简单版结构
如果当前是一个人或小团队做，没必要一开始就上很重的架构。
可以先从这个最小结构开始：
src/
app/
prompt/
agent/
test/
learn/
workspace/

features/
prompt/
agent/
test/
tutorial/
workspace/
search/
ranking/

shared/
ui/
lib/
hooks/
types/
这版已经足够开工。
等项目变大，再补：
● entities
● 更细的 model 分层
● 更明确的 feature 边界
十、一句话建议
这类项目最适合“按业务域组织 + 少量共享层”的结构。
不适合：
● 按页面零散堆代码
● 把所有组件都塞进一个 components/
● 一开始就把搜索、标签、排行硬写死在某个页面里
更合适的是：
● 路由按 5 个一级导航拆
● 功能按业务模块拆
● 通用组件单独放
● 搜索、标签、排行做成横切 feature
● 测试台独立成完整模块
