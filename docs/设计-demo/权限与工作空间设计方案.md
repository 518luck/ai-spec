# 权限与工作空间设计方案

> 状态：设计中 · 暂未落地
>
> 适用范围：个人空间 + 团队空间 的鉴权、成员关系、套餐额度。
>
> 本方案源于一次架构推演：从「想写鉴权高阶函数」出发，依次暴露并解决了权限依据、工作空间实体、成员关系存储、套餐字段归属四个层级的问题。本文是推演结论的固化。

---

## 一、核心原则：三层职责分离

整套设计建立在一个根本判断上——**身份、权限、缓存是三个正交维度，必须物理隔离**。

| 层 | 回答的问题 | 数据载体 | 变更频率 |
|---|---|---|---|
| **身份层** | 你是谁？ | `session.user.id` | 登录后基本不变 |
| **业务层** | 你能干什么？能享受多少配额？ | DB 表（User / Project / ProjectUsers） | 随业务动态变化 |
| **缓存层** | 如何加速查询？ | Redis（kv.ts） | 按策略失效 |

### 关键决策：session 只表示身份

**`session` 不存放任何 role / plan 字段**，理由：

1. **身份凭证应最小化**——JWT 每次请求都跟着走，越轻越好；
2. **业务属性会陈旧**——用户今天升级套餐，JWT 里还写着旧值，要么等 token 过期才生效，要么强制重签；
3. **泄漏面控制**——JWT 被截获时，攻击者只拿到「你是谁」，不暴露套餐和角色。

这是安全设计的基本原则，与 GitHub、Linear、Vercel 等主流产品的做法一致。

## 二、数据模型

### 2.1 表关系总览

```
┌──────────┐       ┌──────────────────┐       ┌──────────┐
│   User   │       │    Project       │       │ Project  │
│ (个人)   │◀──────│    Users         │──────▶│ (团队)   │
│          │  N:N  │  (成员中间表)    │  N:N  │          │
└──────────┘       └──────────────────┘       └──────────┘
     │                                            │
     │ owner_id                                   │ plan
     │                                            │
     ▼                                            ▼
┌──────────────────────────────────────────────────────────┐
│              PromptRecord / PromptDraft / ...            │
│   workspace_id = null  → 个人资源                        │
│   workspace_id = "xxx" → 团队资源                        │
└──────────────────────────────────────────────────────────┘
```

### 2.2 表结构定义

#### User（扩展，加 `plan`）

```prisma
model User {

  free
```

#### Project（新建，团队实体）

```prisma
model Project {
  // ① 主键
  id String @id @default(cuid())

  // ② 业务字段
  name        String
  slug        String?  @unique  // URL 友好标识，可选
  description String?

  // ▼ 团队套餐档位，控制团队空间额度与功能（与 UserPlan 分开，因为付费主体不同）
  plan        ProjectPlan @default(free)

  // ③ 时间戳
  created_at DateTime @default(now())
  updated_at DateTime @updatedAt

  // ④ 外键列
  owner_id String

  // ⑤ 关联关系
  owner       User           @relation("ProjectOwner", fields: [owner_id], references: [id], onDelete: Cascade)
  memberships ProjectUsers[]

  // ⑥ 索引
  @@index([owner_id])
  @@schema("auth")  // 团队归属鉴权体系，与 User 同 schema
}

// 团队套餐枚举（独立于个人套餐，未来团队可能有 team / enterprise 等档位）
enum ProjectPlan {
  free
  team
  enterprise
}
```

#### ProjectUsers（新建，成员中间表）

```prisma
model ProjectUsers {
  // ① 主键
  id String @id @default(cuid())

  // ② 时间戳
  created_at DateTime @default(now())

  // ③ 外键列
  project_id String
  user_id    String

  // ▼ 团队内角色：owner / admin / member / viewer，与全局 UserPlan 完全正交
  role        ProjectRole @default(member)

  // ⑤ 关联关系
  project Project @relation(fields: [project_id], references: [id], onDelete: Cascade)
  user    User    @relation(fields: [user_id], references: [id], onDelete: Cascade)

  // ⑥ 约束：同一用户在同一团队只能有一条记录，防止角色矛盾
  @@unique([project_id, user_id])
  @@index([user_id])  // 加速「查询某用户的所有团队」
  @@schema("auth")
}

// 团队角色枚举
enum ProjectRole {
  owner     // 创建者，不可移除，可解散团队
  admin     // 管理员，可邀请/移除成员、改角色
  member    // 普通成员，可读写团队资源
  viewer    // 只读成员，仅可查看
}
```

### 2.3 字段归属决策依据

| 字段 | 位置 | 为什么不放别处 |
|---|---|---|
| `User.plan` | User 表 | 个人付费、个人享用，跟着用户走 |
| `Project.plan` | Project 表 | 团队付费、团队共享额度，跟着团队走 |
| `ProjectUsers.role` | 中间表 | 同一用户在不同团队角色不同，必须按「用户×团队」一行 |
| `role` 不进 session | — | 业务属性会陈旧，session 只表身份 |

## 三、权限判断流程

### 3.1 两个正交维度

```
请求来了，带 session.user.id + 可选 workspaceId
   │
   ├─ workspaceId 为空 → 个人空间
   │     用 session.user.id 查 User.plan
   │     判断「这个用户能干这件事吗」
   │
   └─ workspaceId 非空 → 团队空间
         用 session.user.id + workspaceId 查 ProjectUsers
         拿到 membership.role，判断「这个角色能干这件事吗」
         同时查 Project.plan，判断「这个团队套餐支持吗」
```

**注意**：空间归属由 `workspace_id` 判断，不由 role 判断。这是两个独立的维度。

### 3.2 完整判断示例

```
李四请求 PATCH /api/projects/团队A/resources/xxx （带 workspaceId=团队A）

① 身份认证（withSession，已有）
   → session.user.id = 李四

② 工作空间解析（withWorkspace，待写）
   → 查 ProjectUsers WHERE user_id=李四 AND project_id=团队A
   → 命中，role=editor
   → 注入 { workspace, membership }

③ 权限校验（withPermission("resource.write")，待写）
   → editor 能 resource.write 吗？能 → 放行

④ 套餐校验（可选，withPlan，未来）
   → 查 Project.plan = "team"
   → team 套餐支持该功能吗？支持 → 放行

⑤ 业务 handler 执行
```

## 四、中间件层（高阶函数）

### 4.1 设计原则

- **不嵌套叠加**：每个高阶函数只注入自己职责内的字段，避免 `withWorkspace(withPermission(withSession(handler)))` 的嵌套地狱；
- **复用认证内核**：`withSession` 已有的认证/限流/错误封装逻辑，将抽成内部 `resolveContext` 函数供其他高阶函数复用（后续重构）；
- **命名遵循规范**：`withXxx` 形式，已写入 `AGENTS.md` 代码风格指南。

### 4.2 高阶函数清单

| 高阶函数 | 职责 | 注入字段 | 状态 |
|---|---|---|---|
| `withSession(handler)` | 身份认证 + 限流 + 错误封装 | `session` | ✅ 已有 |
| `withPermission(action)(handler)` | RBAC 动作校验 | `session` | 📋 待写 |
| `withWorkspace(handler)` | 工作空间解析 + 归属校验 | `session`, `workspace`, `membership` | 📋 待写 |
| `withPlan(metric)(handler)` | 套餐/额度预检（未来） | `session`, `plan`, `quota` | 🔮 未来 |

### 4.3 调用示例

```ts
// 个人空间接口：只校验身份 + 个人 RBAC
export const POST = withPermission("promptDraft.write")(
    async ({ session }) => {
        // 业务逻辑
    }
);

// 团队空间接口：校验身份 + 工作空间归属 + 团队 RBAC
export const POST = withWorkspace(
    withPermission("project.write")(
        async ({ session, workspace, membership }) => {
            // 业务逻辑
        }
    )
);
```

## 五、查询层封装（缓存接入点）

### 5.1 设计原则

**把「查权限/套餐」的逻辑封装成函数**，内部先实时查库。未来加缓存时，**只改函数实现，调用方零改动**。

### 5.2 查询函数清单

```ts
// src/shared/lib/ohs/local/appservice/rbac/abilities.ts（待建）

// 查用户个人套餐
export const getUserPlan = async (userId: string): Promise<UserPlan> => {
    return prisma.user.findUnique({
        where: { id: userId },
        select: { plan: true },
    });
};

// 查用户在某团队的角色（不存在返回 null）
export const getMembership = async (
    projectId: string,
    userId: string,
): Promise<ProjectUsers | null> => {
    return prisma.projectUsers.findUnique({
        where: {
            project_id_user_id: { project_id: projectId, user_id: userId },
        },
        include: { project: { select: { plan: true, name: true } } },
    });
};

// 判断用户能否执行某 action（RBAC 核心）
export const can = (role: ProjectRole, action: PermissionAction): boolean => {
    // 基于 ROLE_PERMISSIONS 表查询
};
```

### 5.3 缓存策略（未来接入）

```ts
// 未来：加缓存层，只改函数内部，调用方零改动
export const getUserPlan = async (userId: string): Promise<UserPlan> => {
    const cacheKey = `user:plan:${userId}`;
    const cached = await kvGet<UserPlan>(cacheKey);
    if (cached) return cached;                    // ← 缓存层就这一段

    const data = await prisma.user.findUnique({   // 原查库逻辑不动
        where: { id: userId },
        select: { plan: true },
    });
    await kvSet(cacheKey, data, 300);             // 回填，TTL 5 分钟
    return data;
};

// 数据变更时主动失效
export const invalidateUserPlan = async (userId: string): Promise<void> => {
    await kvDel(`user:plan:${userId}`);
};
```

#### 缓存对象与失效时机

| 缓存对象 | Key | TTL | 失效时机 |
|---|---|---|---|
| 用户套餐 | `user:plan:{userId}` | 5 分钟 | 用户升级/降级 |
| 团队成员关系 | `membership:{projectId}:{userId}` | 5 分钟 | 加成员/移除/改角色 |
| 团队套餐 | `project:plan:{projectId}` | 5 分钟 | 团队升级/降级 |

#### 缓存模式：cache-aside + write-invalidate

- **TTL 兜底**：防止主动失效漏调时的最终一致（最多 5 分钟陈旧）；
- **主动失效**：保证正常流程下立即可见；
- **不缓存进 session**：避免 JWT 陈旧问题。

## 六、额度扩展（未来）

### 6.1 独立额度表

额度是「高频并发写入 + 原子扣减 + 周期重置」的热路径数据，**必须独立成表**，不污染业务表。

```prisma
model UserQuota {
  user_id      String   @id
  tokens_used  Int      @default(0)
  tokens_limit Int
  period_start DateTime
  period_end   DateTime
  last_deduct_at DateTime?

  user User @relation(fields: [user_id], references: [id], onDelete: Cascade)

  @@schema("quota")
}

model TeamQuota {
  workspace_id String   @id
  tokens_used  Int      @default(0)
  tokens_limit Int
  period_start DateTime
  period_end   DateTime
  last_deduct_at DateTime?

  project Project @relation(fields: [workspace_id], references: [id], onDelete: Cascade)

  @@schema("quota")
}
```

### 6.2 为什么不把额度塞进 User/Project 表

| 维度 | 塞业务表 | 独立额度表（推荐） |
|---|---|---|
| 查询 | 无 JOIN | 多一个主键 JOIN（微秒级） |
| 写入 | 高频写业务表 | 只锁额度表 |
| 锁影响 | 阻塞关系查询 | 业务表无感 |
| 周期重置 | 长事务锁业务表 | 只扫额度表 |

**结论**：用「一个 JOIN」换「零锁竞争」，极其划算。这是 GitHub、Stripe、OpenAI 等的主流做法。

### 6.3 扣减主体规则

| 场景 | 扣谁的额度 |
|---|---|
| 用户在个人空间调接口 | 用户额度（UserQuota） |
| 用户在团队空间调接口 | 团队额度（TeamQuota） |

判断规则一句话：**`workspace_id` 非空扣团队，为空扣用户**。

## 八、设计推演过程（思维链条）

> 本方案不是先验设计，而是从需求出发被约束推着长出来的。完整推演如下：

```
① 起点：想写鉴权高阶函数
   └─ 目标：给 API route 加鉴权中间件，抽离样板代码
      │
      ▼
② 触发问题：写 withPermission 时发现「判断权限需要 role，session 里没有」
   └─ 暴露：权限的依据从哪来？
      │
      ▼
③ 深入权限：role 在哪？进 session 还是查库？
   └─ 发现：权限其实分两种——
      │  • 个人空间权限（全局 plan / role）
      │  • 团队空间权限（team role，按团队不同）
      │
      ▼
④ 触发问题：「团队空间的权限」是什么？用户怎么和团队绑定？
   └─ 暴露：工作空间（团队）概念缺失
      │
      ▼
⑤ 设计工作空间：团队怎么组织？
   └─ 决策：
      │  • 个人/团队拆开，不混合
      │  • 区分用 workspace_id，不用 role
      │  • 团队需要实体表 + 成员关系表
      │
      ▼
⑥ 触发问题：成员关系怎么存？能不能塞 Project 表？
   └─ 发现：多对多关系 + 按对带属性（角色）
      │  • 塞 Project 表字段 = 锁竞争、查询慢（反模式）
      │  • 必须独立中间表
      │
      ▼
⑦ 设计 ProjectUsers 中间表：每对「用户-团队」一行
   └─ 决策：
      │  • ProjectUsers 独立成表
      │  • (project_id, user_id) 加唯一约束
      │  • role 字段存团队内角色
      │
      ▼
⑧ 触发问题：那「付费/免费」这种全局属性放哪？
   └─ 关键洞察：
      │  • session 只表身份，不掺 plan/role（职责分离）
      │  • plan 挂业务表（User.plan 或 Project.plan）
      │  • 实时查库，不缓存进 session
      │
      ▼
⑨ 最终架构：三层正交设计
   ├─ 身份层：session 只存 id
   ├─ 业务层：User.plan / Project / ProjectUsers 分开存
   └─ 查询层：封装函数，实时查库（未来加缓存）
      │
      ▼
⑩ 延伸：缓存策略（cache-aside + write-invalidate）
   └─ 查询函数内部加缓存，调用方零改动
```

## 九、关键决策记录（ADR 总结）

| # | 决策 | 理由 | 替代方案（被否） |
|---|---|---|---|
| 1 | session 不存 role/plan | 身份凭证最小化，避免陈旧 | role 进 JWT（数据会陈旧） |
| 2 | 个人/团队用 workspace_id 区分 | 与权限维度正交 | 用 role 区分（混淆维度） |
| 3 | 个人空间不进 workspace 表 | 用 owner_id 即可，无协作需求 | GitHub 式统一模型（冗余） |
| 4 | 成员关系独立成 ProjectUsers 表 | 多对多关系的物理要求 | 塞 Project 表字段（锁竞争） |
| 5 | 权限定义留在代码（permissions.ts） | 与代码逻辑强绑定 | 进数据库（不一致风险） |
| 6 | 实时查库，不缓存进 session | 数据一致性最好 | 缓存进 JWT（陈旧风险） |
| 7 | 额度独立成表 | 高频写入不能污染业务表 | 塞 Project 表（锁竞争） |
| 8 | 缓存作为查询函数实现细节 | 调用方零改动，可加可撤 | 在中间件层缓存（耦合） |
