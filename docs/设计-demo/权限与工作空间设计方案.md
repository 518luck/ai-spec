# 权限与工作空间设计方案

> 状态：schema 已落地（Team / TeamMember / User.plan + 三套 enum），中间件层待写
>
> 适用范围：个人空间 + 团队空间 的鉴权、成员关系、套餐额度。
>
> 命名说明：第八、九章的「推演过程」与「决策记录」保留讨论时的原始术语（Project / ProjectUsers），最终落地命名为 **Team / TeamMember**，对应关系见第二章 schema 定义。
>
> 本方案源于一次架构推演：从「想写鉴权高阶函数」出发，依次暴露并解决了权限依据、工作空间实体、成员关系存储、套餐字段归属四个层级的问题。本文是推演结论的固化。

## 一、核心原则：三层职责分离

整套设计建立在一个根本判断上——**身份、权限、缓存是三个正交维度，必须物理隔离**。

| 层 | 回答的问题 | 数据载体 | 变更频率 |
|---|---|---|---|
| **身份层** | 你是谁？ | `session.user.id` | 登录后基本不变 |
| **业务层** | 你能干什么？能享受多少配额？ | DB 表（User / Team / TeamMember） | 随业务动态变化 |
| **缓存层** | 如何加速查询？ | Redis（kv.ts） | 按策略失效 |

### 关键决策：session 只表示身份

**`session` 不存放任何 role / plan 字段**，理由：

1. **身份凭证应最小化**——JWT 每次请求都跟着走，越轻越好；
2. **业务属性会陈旧**——用户今天升级套餐，JWT 里还写着旧值，要么等 token 过期才生效，要么强制重签；
3. **泄漏面控制**——JWT 被截获时，攻击者只拿到「你是谁」，不暴露套餐和角色。

这是安全设计的基本原则，与 GitHub、Linear、Vercel 等主流产品的做法一致。

## 二、数据模型

### 2.1 表关系总览

```
┌──────────┐       ┌──────────────────┐       ┌──────────┐
│   User   │       │   TeamMember     │       │   Team   │
│ (个人)   │◀──────│  (成员中间表)    │──────▶│ (团队)   │
│          │  N:N  │  role: 角色      │  N:N  │  plan    │
└──────────┘       └──────────────────┘       └──────────┘
                        │   唯一连接通道
                        ▼
              User 与 Team 之间无直连字段，
              创建者由 TeamMember 中 role=owner 的行表达
```

> 资源归属（PromptRecord / PromptDraft / ...）由 `workspace_id` 判断：
> `null` → 个人资源；非空 → 团队资源。

### 2.2 表结构定义

#### User（扩展，加 `plan`）

```prisma
// 用户本人信息
model User {
  id            String    @id @default(cuid())
  name          String?
  email         String    @unique
  emailVerified DateTime?
  image         String?

  // 密码和安全字段
  passwordHash         String?
  invalidLoginAttempts Int       @default(0)
  lockedAt             DateTime?

  // 个人空间套餐档位，控制个人额度与付费功能；不进 session，每次实时查库
  plan UserPlan @default(free)

  accounts Account[]
  sessions Session[]
  tokens   Token[]

  // 提示词反向关系
  promptRecords        PromptRecord[]        @relation("PromptRecordOwner")
  promptRecordEdited   PromptRecord[]        @relation("PromptRecordEditor")
  promptRecordVersions PromptRecordVersion[] @relation("VersionEditor")
  promptDrafts         PromptDraft[]         @relation("PromptDraftOwner")
  promptFavorites      PromptFavorite[]

  // 团队反向关系：该用户加入的所有团队，创建者身份由其中 role=owner 的行表达
  memberships TeamMember[]

  @@schema("auth")
}

// 个人空间套餐枚举：free=免费，pro=付费个人版
enum UserPlan {
  free
  pro

  @@schema("auth")
}
```

#### Team（新建，团队实体）

```prisma
// 团队实体：多人协作的工作空间，承载团队级资源与额度
model Team {
  // ① 主键
  id String @id @default(cuid())

  // ② 业务字段
  name        String
  slug        String? @unique // URL 友好标识，用于团队专属页路径
  description String?
  plan        TeamPlan @default(free) // 团队套餐档位，控制团队空间额度与付费功能

  // ③ 时间戳
  created_at DateTime @default(now())
  updated_at DateTime @updatedAt

  // ④ 关联关系
  memberships TeamMember[] // 团队的全部成员关系；创建者由其中 role=owner 的行表达

  // ⑤ 索引
  @@schema("auth")
}

// 团队套餐枚举：free=免费团队，team=付费团队版，enterprise=企业版
enum TeamPlan {
  free
  team
  enterprise

  @@schema("auth")
}
```

#### TeamMember（新建，成员中间表）

```prisma
// 团队成员关系：每对「用户×团队」一行，承载该用户在此团队内的角色
model TeamMember {
  // ① 主键
  id String @id @default(cuid())

  // ② 业务字段
  role TeamRole @default(member)

  // ③ 时间戳
  created_at DateTime @default(now())

  // ④ 外键列
  team_id String
  user_id String

  // ⑤ 关联关系
  team Team @relation(fields: [team_id], references: [id], onDelete: Cascade)
  user User @relation(fields: [user_id], references: [id], onDelete: Cascade)

  // ⑥ 索引
  @@unique([team_id, user_id]) // 同一用户在同一团队只能有一条记录，防止角色矛盾
  @@index([user_id]) // 按用户查询其加入的全部团队
  @@schema("auth")
}

// 团队角色枚举：owner=创建者（不可移除），admin=管理员，member=普通成员，viewer=只读
enum TeamRole {
  owner
  admin
  member
  viewer

  @@schema("auth")
}
```

### 2.3 字段归属决策依据

| 字段 | 位置 | 为什么不放别处 |
|---|---|---|
| `User.plan` | User 表 | 个人付费、个人享用，跟着用户走 |
| `Team.plan` | Team 表 | 团队付费、团队共享额度，跟着团队走 |
| `TeamMember.role` | 中间表 | 同一用户在不同团队角色不同，必须按「用户×团队」一行 |
| `role` 不进 session | — | 业务属性会陈旧，session 只表身份 |

## 三、统一 Action 体系（核心架构）

> **架构原则**：Action 是唯一真相源，API Key scope 和 Web 端 RBAC 都是它的「视图」。
>
> **业界印证**：OpenAI RBAC（组织角色 + API Key scope 共用 permission）、Go-rbac（角色中间件 + scope 中间件串联）、WorkOS RBAC（permission 为最细粒度）均采用此模型。
>
> **演进背景**：本项目曾设计为「API Key scope 与团队 RBAC 两套独立体系」，但因「加新资源要改两处、概念割裂」等冗余问题，经业界调研后统一为本方案。

### 3.1 核心思路：一份 action，两种授予方式

```
              统一的 Action 清单（actions.ts，唯一真相源）
              promptRecord.read / promptRecord.write / ...
              project.read / project.write / ...
                       │
           ┌───────────┴───────────┐
           ▼                       ▼
     Web 端权限                 API Key 权限
     来源：UserPlan +           来源：token.scopes
          TeamMember.role       （用户创建 key 时勾选的 action 子集）
           │                       │
           ▼                       ▼
     角色展开为 action 集合     直接是 action 子集
     （查 TEAM_ROLE_ACTIONS）
```

**关键认知**：

- **action 是统一概念**——无论个人资源还是团队资源，都是「在某个资源上做一个动作」，只是资源的归属不同；
- **两种授予来源共用同一份 action**——Web 端通过角色间接映射，API Key 直接列 scope，但两者查的是同一套定义；
- **API Key 物理上拿不到团队 action**——通过 `apiKeyGrantable` 元信息从源头禁止，创建 key 时勾选不到，后端校验也拒。

### 3.2 Action 元信息：区分空间归属与可授予性

每个 action 带两个关键标记：

| 标记 | 含义 | 取值 |
|------|------|------|
| `space` | 归属哪个空间 | `"personal"` / `"team"` |
| `apiKeyGrantable` | API Key 创建时能否勾选 | `true` / `false` |

这两者的关系：**个人资源的 action 通常 `apiKeyGrantable: true`，团队资源的 action 一律 `apiKeyGrantable: false`**（因为团队资源不允许 SDK / MCP 调用）。

```ts
// actions.ts 的形态（待落地）
type ActionDef = {
    action: string;                    // "promptRecord.write"
    space: "personal" | "team";        // 归属空间
    apiKeyGrantable: boolean;          // API Key 能否授予
    description: string;               // 业务说明
};

export const ACTION_DEFS: readonly ActionDef[] = [
    // 个人资源 - API Key 可授予
    { action: "promptRecord.read",  space: "personal", apiKeyGrantable: true,  description: "浏览提示词收录" },
    { action: "promptRecord.write", space: "personal", apiKeyGrantable: true,  description: "编辑提示词收录" },
    // ... 其他个人资源（共 7 资源 × 读写 = 14 个）

    // 团队资源 - API Key 不可授予
    { action: "project.read",       space: "team", apiKeyGrantable: false, description: "查看团队项目" },
    { action: "project.write",      space: "team", apiKeyGrantable: false, description: "管理团队项目" },
    // ... 其他团队资源（project/member/team/security/logs）
];

// 派生：所有 action 字符串
export const ACTIONS = ACTION_DEFS.map((d) => d.action) as const;
export type Action = (typeof ACTIONS)[number];

// 派生：API Key 可授予的 scope（过滤掉团队 action）
export const API_KEY_SCOPES = ACTION_DEFS
    .filter((d) => d.apiKeyGrantable)
    .map((d) => d.action);
```

### 3.3 文件分工与派生关系

```
actions.ts          ← 唯一真相源：所有 action + 元信息（space, apiKeyGrantable）
    │
    ├─→ scopes.ts              API Key 的 scope 体系
    │     SCOPES = [...API_KEY_SCOPES, "apis.read", "apis.all"]
    │     （团队 action 自动被过滤，因为 apiKeyGrantable=false）
    │     buildScopes / getScopesForResource / scopesToName 等保留
    │
    └─→ permissions.ts         Web 端的角色 → action 映射
          TEAM_ROLE_ACTIONS: Record<TeamRole, readonly Action[]>
          （团队 RBAC，落地时填具体映射）
```

**加新资源时只改 `actions.ts` 一处**——`scopes.ts` 和 `permissions.ts` 自动同步。

### 3.4 权限判断流程

```
请求来了，带 session.user.id + 可选 workspaceId
   │
   ├─ 携带 Authorization: Bearer（API Key 调用）
   │     → withSession 解析 token.scopes
   │     → 判断「token.scopes 是否包含目标 action」
   │     → 团队 action 因 apiKeyGrantable=false 永远不在 scope 里，天然拒绝
   │
   └─ 无 Bearer（Web 端 session）
         ├─ workspaceId 为空 → 个人空间
         │     → 查 UserPlan（实时查库）
         │     → 判断「这个用户能干这个 action 吗」
         │
         └─ workspaceId 非空 → 团队空间
               → 查 TeamMember 拿 role
               → 查 TEAM_ROLE_ACTIONS[role] 是否包含目标 action
               → 查 Team.plan 判断套餐是否支持
```

**注意**：空间归属由 `workspace_id` 判断，action 的 `space` 标记是「归属声明」，两者语义对齐但不互相决定。

### 3.5 完整判断示例

```
李四请求 PATCH /api/teams/团队A/projects/xxx （带 workspaceId=团队A）

① 身份认证（withSession，已有）
   → session.user.id = 李四

② 工作空间解析（withWorkspace，待写）
   → 查 TeamMember WHERE user_id=李四 AND team_id=团队A
   → 命中，role=admin
   → 注入 { workspace, membership }

③ 权限校验（withPermission("project.write")，待写）
   → action="project.write"，space="team"
   → 查 TEAM_ROLE_ACTIONS[admin] 是否包含 "project.write"
   → 包含 → 放行

④ 套餐校验（可选，withPlan，未来）
   → 查 Team.plan = "team"
   → team 套餐支持该功能吗？支持 → 放行

⑤ 业务 handler 执行
```

## 四、中间件层（高阶函数）

### 4.1 设计原则

- **不嵌套叠加**：每个高阶函数只注入自己职责内的字段，避免 `withWorkspace(withPermission(withSession(handler)))` 的嵌套地狱；
- **复用认证内核**：`withSession` 已有的认证/限流/错误封装逻辑，将抽成内部 `resolveContext` 函数供其他高阶函数复用（后续重构）；
- **命名遵循规范**：`withXxx` 形式，已写入 `AGENTS.md` 代码风格指南。

### 4.2 高阶函数清单

| 高阶函数 | 职责 | 注入字段 | 状态 |
|---|---|---|---|
| `withSession(handler)` | 身份认证 + 限流 + 错误封装 | `session` | ✅ 已有 |
| `withPermission(action)(handler)` | RBAC 动作校验 | `session` | 📋 待写 |
| `withWorkspace(handler)` | 工作空间解析 + 归属校验 | `session`, `workspace`, `membership` | 📋 待写 |
| `withPlan(metric)(handler)` | 套餐/额度预检（未来） | `session`, `plan`, `quota` | 🔮 未来 |

### 4.3 调用示例

```ts
// 个人空间接口：只校验身份 + 个人 RBAC
export const POST = withPermission("promptDraft.write")(
    async ({ session }) => {
        // 业务逻辑
    }
);

// 团队空间接口：校验身份 + 工作空间归属 + 团队 RBAC
export const POST = withWorkspace(
    withPermission("project.write")(
        async ({ session, workspace, membership }) => {
            // 业务逻辑
        }
    )
);
```

## 五、查询层封装（缓存接入点）

### 5.1 设计原则

**把「查权限/套餐」的逻辑封装成函数**，内部先实时查库。未来加缓存时，**只改函数实现，调用方零改动**。

### 5.3 缓存策略（未来接入）

```ts
// 未来：加缓存层，只改函数内部，调用方零改动
export const getUserPlan = async (userId: string): Promise<UserPlan> => {
    const cacheKey = `user:plan:${userId}`;
    const cached = await kvGet<UserPlan>(cacheKey);
    if (cached) return cached;                    // ← 缓存层就这一段

    const data = await prisma.user.findUnique({   // 原查库逻辑不动
        where: { id: userId },
        select: { plan: true },
    });
    await kvSet(cacheKey, data, 300);             // 回填，TTL 5 分钟
    return data;
};

// 数据变更时主动失效
export const invalidateUserPlan = async (userId: string): Promise<void> => {
    await kvDel(`user:plan:${userId}`);
};
```

#### 缓存对象与失效时机

| 缓存对象 | Key | TTL | 失效时机 |
|---|---|---|---|
| 用户套餐 | `user:plan:{userId}` | 5 分钟 | 用户升级/降级 |
| 团队成员关系 | `membership:{teamId}:{userId}` | 5 分钟 | 加成员/移除/改角色 |
| 团队套餐 | `team:plan:{teamId}` | 5 分钟 | 团队升级/降级 |

#### 缓存模式：cache-aside + write-invalidate

- **TTL 兜底**：防止主动失效漏调时的最终一致（最多 5 分钟陈旧）；
- **主动失效**：保证正常流程下立即可见；
- **不缓存进 session**：避免 JWT 陈旧问题。

## 六、额度扩展（未来）

### 6.1 独立额度表

额度是「高频并发写入 + 原子扣减 + 周期重置」的热路径数据，**必须独立成表**，不污染业务表。

```prisma
model UserQuota {
  user_id      String   @id
  tokens_used  Int      @default(0)
  tokens_limit Int
  period_start DateTime
  period_end   DateTime
  last_deduct_at DateTime?

  user User @relation(fields: [user_id], references: [id], onDelete: Cascade)

  @@schema("quota")
}

model TeamQuota {
  team_id      String   @id
  tokens_used  Int      @default(0)
  tokens_limit Int
  period_start DateTime
  period_end   DateTime
  last_deduct_at DateTime?

  team Team @relation(fields: [team_id], references: [id], onDelete: Cascade)

  @@schema("quota")
}
```

### 6.2 为什么不把额度塞进 User/Team 表

| 维度 | 塞业务表 | 独立额度表（推荐） |
|---|---|---|
| 查询 | 无 JOIN | 多一个主键 JOIN（微秒级） |
| 写入 | 高频写业务表 | 只锁额度表 |
| 锁影响 | 阻塞关系查询 | 业务表无感 |
| 周期重置 | 长事务锁业务表 | 只扫额度表 |

**结论**：用「一个 JOIN」换「零锁竞争」，极其划算。这是 GitHub、Stripe、OpenAI 等的主流做法。

### 6.3 扣减主体规则

| 场景 | 扣谁的额度 |
|---|---|
| 用户在个人空间调接口 | 用户额度（UserQuota） |
| 用户在团队空间调接口 | 团队额度（TeamQuota） |

判断规则一句话：**`workspace_id` 非空扣团队，为空扣用户**。

## 八、设计推演过程（思维链条）

> 本方案不是先验设计，而是从需求出发被约束推着长出来的。完整推演如下：

```
① 起点：想写鉴权高阶函数
   └─ 目标：给 API route 加鉴权中间件，抽离样板代码
      │
      ▼
② 触发问题：写 withPermission 时发现「判断权限需要 role，session 里没有」
   └─ 暴露：权限的依据从哪来？
      │
      ▼
③ 深入权限：role 在哪？进 session 还是查库？
   └─ 发现：权限其实分两种——
      │  • 个人空间权限（全局 plan / role）
      │  • 团队空间权限（team role，按团队不同）
      │
      ▼
④ 触发问题：「团队空间的权限」是什么？用户怎么和团队绑定？
   └─ 暴露：工作空间（团队）概念缺失
      │
      ▼
⑤ 设计工作空间：团队怎么组织？
   └─ 决策：
      │  • 个人/团队拆开，不混合
      │  • 区分用 workspace_id，不用 role
      │  • 团队需要实体表 + 成员关系表
      │
      ▼
⑥ 触发问题：成员关系怎么存？能不能塞 Project 表？
   └─ 发现：多对多关系 + 按对带属性（角色）
      │  • 塞 Project 表字段 = 锁竞争、查询慢（反模式）
      │  • 必须独立中间表
      │
      ▼
⑦ 设计 ProjectUsers 中间表：每对「用户-团队」一行
   └─ 决策：
      │  • ProjectUsers 独立成表
      │  • (project_id, user_id) 加唯一约束
      │  • role 字段存团队内角色
      │
      ▼
⑧ 触发问题：那「付费/免费」这种全局属性放哪？
   └─ 关键洞察：
      │  • session 只表身份，不掺 plan/role（职责分离）
      │  • plan 挂业务表（User.plan 或 Project.plan）
      │  • 实时查库，不缓存进 session
      │
      ▼
⑨ 最终架构：三层正交设计
   ├─ 身份层：session 只存 id
   ├─ 业务层：User.plan / Project / ProjectUsers 分开存
   └─ 查询层：封装函数，实时查库（未来加缓存）
      │
      ▼
⑩ 延伸：缓存策略（cache-aside + write-invalidate）
   └─ 查询函数内部加缓存，调用方零改动
```

## 九、关键决策记录（ADR 总结）

| # | 决策 | 理由 | 替代方案（被否） |
|---|---|---|---|
| 1 | session 不存 role/plan | 身份凭证最小化，避免陈旧 | role 进 JWT（数据会陈旧） |
| 2 | 个人/团队用 workspace_id 区分 | 与权限维度正交 | 用 role 区分（混淆维度） |
| 3 | 个人空间不进 Team 表 | 个人资源用 PromptRecord.owner_id 判断归属即可，无协作需求 | GitHub 式统一模型（冗余） |
| 4 | 成员关系独立成 ProjectUsers 表 | 多对多关系的物理要求 | 塞 Project 表字段（锁竞争） |
| 5 | 权限定义留在代码（permissions.ts） | 与代码逻辑强绑定 | 进数据库（不一致风险） |
| 6 | 实时查库，不缓存进 session | 数据一致性最好 | 缓存进 JWT（陈旧风险） |
| 7 | 额度独立成表 | 高频写入不能污染业务表 | 塞 Project 表（锁竞争） |
| 8 | 缓存作为查询函数实现细节 | 调用方零改动，可加可撤 | 在中间件层缓存（耦合） |
| 9 | Team 不设 owner_id 直连字段 | TeamMember.role=owner 即创建者，单一真相源避免双写不一致 | Team.owner_id 直连（多一处同步点） |
| 10 | 砍掉 ROLE 中间层 | API Key 鉴权直接看 token.scopes，无需角色翻译；原 ROLE 是未理清业务时 AI 随意分配的，不反映真实约束 | 保留 ROLE 做 scope 套餐预设（虚假抽象） |
| 11 | 统一 Action 体系（方案 B） | action 是唯一真相源，API Key scope 和 Web 端 RBAC 共用同一份 action；加新资源改一处，概念统一 | 两套独立体系（PersonalAction + TeamAction 重复维护） |
| 12 | API Key 用 `apiKeyGrantable` 隔离团队 action | 团队资源不允许 SDK 调用，从 action 元信息源头禁止授予，创建 key 时勾选不到 | 两份独立 action 清单（重复且易漏同步） |
