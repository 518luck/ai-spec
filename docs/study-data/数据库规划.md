# 数据库规划

不建议你一开始就上“很重的 RBAC”，但建议从第一天就按 RBAC-compatible 的方式设计权限模型。

## 第一阶段：先做最小权限模型

你至少要有这几类能力区分：

- 游客：可浏览公开 Prompt / Agent / 教程
- 登录用户：可收藏、建文件夹、保存草稿、记录测试历史
- 内容作者：可创建、编辑自己的内容
- 管理员：可审核、下架、推荐、改标签、处理举报

这时候其实更接近：

- 认证：你是谁
- 归属判断：这是不是你的内容
- 状态权限：内容是公开、私有、草稿还是待审核

也就是说，先用 “角色 + 资源归属 + 内容状态”，不要一上来做复杂权限矩阵。

你这个项目里，真正重要的权限通常不是“按钮能不能看见”，而是这些规则：

- 用户只能编辑自己的 prompt / agent / draft / folder
- 私有内容只有 owner 自己能看
- 已发布内容普通用户可读，但不可改
- 管理员可审核、置顶、封禁、改推荐位
- 测试记录默认仅自己可见，除非主动分享

这套规则，用轻量 RBAC 就够了。

## 更具体一点：建议你做“轻 RBAC + ownership”

可以定义很少的角色：

- guest
- user
- creator
- admin

但真正判定时不要只看角色，还要看资源上下文：

- canReadPrompt(user, prompt)
- canEditPrompt(user, prompt)
- canDeleteFolder(user, folder)
- canViewTestSession(user, session)

判断依据一般是：

- 角色
- ownerId
- visibility
- status

例如 Prompt 可以有这些字段：

- authorId
- visibility: public | private | unlisted
- status: draft | published | archived | pending_review

这样你后面要不要扩成完整 RBAC 都不痛苦。

## 什么时候你才需要真正的 RBAC

如果后面出现这些场景，再升级：

- 有运营、编辑、审核员、超级管理员等后台角色
- 不同角色负责不同资源域
- 有组织/团队空间，一个资源属于团队而不是个人
- 有细粒度后台权限，比如“可审核教程但不能管理用户”
- 有企业版、多租户、团队协作

到了这一步，再做完整 RBAC 才值。

你这个项目如果未来会做团队工作区，其实更可能走向：

- RBAC + Workspace Membership
- 甚至 RBAC + ABAC

> ABAC 是 Attribute-Based Access Control，中文一般叫基于属性的访问控制。
> RBAC 是 Role-Based Access Control，中文是基于角色的访问控制。

例如：

- 在某个 workspace 内是 owner / editor / viewer
- 在全站层面又有 admin / moderator

这是后话，不必现在就做满。

## 结合你当前架构，权限放哪

按 FSD 思路，权限不要散落在页面组件里。建议：

- entities/user：用户角色、身份信息
- entities/prompt / entities/agent：资源状态、owner、visibility
- shared/lib/auth：登录态获取
- shared/lib/permission 或 shared/lib/access：权限判断函数
- 页面和 feature 只调用判断结果，不自己拼规则

例如不要在组件里写很多：

```ts
if (user.role === 'admin' || prompt.authorId === user.id) ...
```

而是统一封装成：

```ts
canEditPrompt(user, prompt);
canManageTag(user);
canViewWorkspace(user, workspace);
```

这样后面权限规则变化时，不会炸全项目。

# 什么是真正的RABC 和ABAC

- 明确的“角色 -> 动作”权限矩阵
- 明确的“资源 -> 状态 -> 可执行动作”规则表
- 数据库层面的角色表 / 关联表设计
- 代码层统一的权限判断入口
